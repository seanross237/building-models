## Lesson

When a follow-on mission narrows an older mechanism to one role or feature, ask first whether the narrowing actually produces a smaller faithful theorem object.

If the answer is no, the mission is not a fresh object-definition problem. It is an inherited-object problem, and predecessor negatives may already transfer.

The same warning applies when a follow-on mission keeps the same object family but merely adds new observables or sharper notation. If the support, quotient conventions, and canonization burden are unchanged, new bookkeeping does not reopen Phase 1.

## Evidence

- **exact-ns-tiny-trigger-firewall exploration-001** — The mission tried to isolate Tao's tiny trigger as a standalone exact-NS target. That failed immediately: "tiny," "delayed," and "decisive" only make sense relative to the same carrier / clock / conduit / output chain as before. The faithful object was therefore the predecessor's five-role hypergraph with `a3` distinguished, not a smaller object.
- Once the object collapse was recognized, the earlier singleton non-embeddability result transferred directly. No new support search was justified.
- The only backup was the same non-canonical packet family already known from the predecessor mission. The right terminal verdict was therefore a transferred model-level negative, not a new definition campaign.
- **exact-ns-helical-sign-bottleneck exploration-001** — The mission did not shrink or canonize the old packet family. It added new observables (`SD_part`, `SD_target`) on top of the same support/refinement/witness freedoms. So the predecessor packet negative already constrained the mission: the right first question was still canonicity, not sign optimization.

## Goal Pattern

Before launching a narrowed mechanism mission:

1. Ask: **What is the smallest exact object in which the narrowed feature is still meaningful?**
2. Write the full dependency chain that gives the narrowed feature its meaning.
3. Compare that exact object, and its canonization burden, to the predecessor object.
4. If the role table, support, interaction graph, or canonization burden are unchanged, treat the mission as inherited from the predecessor.
5. If the only novelty is a new observable on the same non-canonical family, run the canonicity screen before authorizing any Phase 1 computation.
6. Check whether predecessor negatives already apply before authorizing fresh computation or construction work.

## Why This Matters

Narrowing can create fake progress. A mission that sounds smaller may still require the entire old object once words like "trigger," "delay," "witness," or "small compared to" are made exact. If the faithful support has not shrunk, the strategizer should spend the turn transferring prior negatives, not pretending a new Phase 1 object exists.

The same failure mode appears when a mission sounds sharper because it introduces better notation or more refined observables. If those observables still live on the same unresolved object family, the mission is inherited even without literal narrowing.
