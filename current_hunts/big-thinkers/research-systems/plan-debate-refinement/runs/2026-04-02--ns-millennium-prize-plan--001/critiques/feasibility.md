# Feasibility Critique

This plan is not yet executable as a research program for the Navier-Stokes Millennium problem. It reads like a taxonomy of promising internal objects, but it does not yet establish that those objects are well-defined, reusable outside the status file, or connected tightly enough to the actual regularity problem to justify the proposed allocation of time.

## Strongest Objections

1. The plan is built around a private object language that is not yet shown to be mathematically stable. Terms like `frozen one-bridge ledger`, `template-defect near-closure`, `leakage sheet`, and `shell-crossing exact triads` may be useful in the internal note system, but feasibility depends on whether they can be turned into canonical definitions that independent mathematicians would accept. Right now that is an assumption, not a result.

2. The early routes mostly target exact identities and local obstruction statements on curated objects. Even if those are proved, they do not automatically translate into a regularity theorem, a blowup exclusion theorem, or a compactness-rigidity contradiction. The plan risks producing refined mechanism evidence that remains one logical layer away from the actual Millennium target.

3. Routes 1 and 2 appear to share the same hidden dependency: if the ledger formalism is not canonical, both routes collapse together. That means the ranking is less diverse than it looks. The plan is treating correlated failure modes as independent workstreams.

4. Route 5 is the only route that resembles a genuine global proof bridge, but it is explicitly deferred until after a canonical extraction lemma exists. That makes the plan circular: the hardest missing ingredient is not downstream of the program, it is the prerequisite for the only route that resembles a real endgame.

5. The sidecar search is too loosely bounded to be useful as a feasibility hedge. The allowed directions are broad enough to consume time, but the success criteria are too weak to force decisive progress. This is exactly how a plan drifts into novelty hunting while claiming discipline.

6. The plan assumes the current-status file is authoritative enough to rank the search space, but it does not show how the stated frontier picture was independently validated. If that status file is wrong or stale in any one place, the entire route ordering becomes unstable.

## Missing Assumptions Or Prerequisites

1. There is no explicit definition package for the core objects. Before any proof work, the plan needs precise statements for every promoted object, the comparison currency, the refinement rule, and the exact meaning of canonicity.

2. There is no evidence that the proposed objects are formalizable in a standard theorem-proving or proof-checking environment, or even in a clean handwritten proof architecture. For a problem at this scale, informal ledger arithmetic is not enough.

3. The plan does not state who is doing the work, what mathematical background is available, or what time budget is being assumed. A feasible Millennium attempt needs a realistic estimate of required expertise and calendar time, not just route names.

4. The plan never states the minimum theorem statements that would count as route success in a way that is non-circular. For example, "exact family theorem" and "firewall lemma" are labels, not success criteria unless the exact hypotheses and conclusions are specified.

5. The plan assumes that exact object-definition progress is the right abstraction, but it does not justify why those objects should control the known obstruction mechanisms for 3D Navier-Stokes. That is a major mathematical premise, not a minor implementation detail.

6. The plan does not separate what is already known from what still needs invention. Without a dependency graph, it is impossible to tell which steps are theorem-proving tasks and which steps are research bets.

## Concrete Revisions

1. Replace the current route list with a dependency graph that names the exact definitions, exact lemmas, and exact failure points. Each route should start with a formal statement, not an object nickname.

2. Add a mandatory validation phase before Route 1 or Route 2: prove that the ledger formalism is canonical under relabeling, refinement, and comparison across equivalent representations. If that fails, stop immediately and do not spend more time on ledger-level theorems.

3. Downgrade Routes 1 and 2 from primary search tracks to test cases for the formalism. They should only remain active if they are shown to imply a nontrivial obstruction or extraction statement relevant to regularity.

4. Move the canonical extraction lemma to the front of the plan, not Route 5. If it cannot be produced early, the rest of the program should be re-scoped around a different proof architecture.

5. Tighten the sidecar search to one lemma target per direction with explicit pass/fail criteria that are strong enough to kill the route quickly. Otherwise the sidecar is just a parallel literature sink.

6. Add a clear resource and timeline budget. The plan should say what can be checked in days, what can be checked in months, and what would require a major theorem-development campaign.

## Salvageability

This is salvageable without a full rewrite, but only if the first half of the plan is rebuilt around formal definitions and dependency structure. As written, the plan is too speculative to execute efficiently, because its most important prerequisites are implicit and its strongest routes are still housed inside a not-yet-validated object framework.

The right repair is not to make the plan more ambitious. It is to make it narrower, more formal, and more front-loaded on canonicity and extraction. If those checks fail, the plan should be reclassified as frontier-mapping rather than a genuine proof campaign.
