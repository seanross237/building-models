# Final Plan: Navier-Stokes Millennium Prize Attempt

## Core Judgment

The draft was right about one thing: this cannot be another generic
estimate-chasing program. It must be theorem-first, exact, and willing to kill
routes quickly.

But the draft was too portfolio-shaped and too ledger-centered. The revised
plan therefore has:

- one main route,
- one fallback route,
- one gating dependency stack,
- and hard stop conditions that close the program rather than letting it drift.

The frozen template-defect and leakage objects remain worth keeping, but only as
subordinate test theorems inside a standard Navier-Stokes contradiction route,
not as ends in themselves.

## Accepted Theorem Targets

### Primary target

Exclude a symmetry-normalized minimal blowup element in an accepted critical
host space, defaulting to `L^3`, by proving:

`minimal blowup element => canonical extraction of one exact one-bridge ledger => canonical obstruction theorem on that ledger => contradiction`

This is the only active route that is directly aimed at a recognized
compactness-rigidity endgame for 3D incompressible Navier-Stokes regularity.

### Fallback target

Prove a new conditional regularity theorem in standard PDE language by deriving
an accepted vorticity-direction regularity criterion from a quantified geometric
hypothesis, rather than from packet or ledger artifacts. The active fallback is:

- combined strain-alignment plus vorticity-direction regularity,
- aimed at a Constantin-Fefferman-type regularity consequence.

This is not the prize proof, but it is a real theorem target on accepted
Navier-Stokes objects and criteria.

## Route Structure

### Main route: extraction-first critical-element program

This is the only primary route.

The frozen packet screens from the status file stay alive only if they can be
attached to this bridge.

### Fallback route: direct geometric regularity theorem

This becomes active only if the main route fails an early gate, or if the main
route cannot state a standard-PDE implication cleanly.

## Dependency-First Plan

### Phase 0: Freeze definitions, provenance, and comparison currency

Before any theorem push, freeze one definition package that maps every promoted
object back to the status-file frontier terms:

- canonical one-bridge ledger,
- template-defect screen,
- leakage screen,
- hostile family,
- harmless refinement,
- relabeling invariance,
- same-currency comparison,
- canonical extraction.

Required deliverables:

- one glossary in standard mathematical prose,
- one provenance map saying where each object comes from,
- one reconstruction rule showing the bookkeeping is recoverable from source
  data rather than hand-maintained.

Hard stop:

- if the objects cannot be stated without hidden witness choice, hidden desired
  channel annotation, or non-reconstructible bookkeeping, stop ledger work and
  move to the fallback route.

### Phase 1: State the exact bridge to an accepted Navier-Stokes theorem target

Write the precise implication chain in standard PDE language:

1. choose the critical host space, default `L^3` unless another accepted space
   is better justified;
2. state the canonical extraction lemma that would attach an exact one-bridge
   ledger to a symmetry-normalized minimal blowup element;
3. state the obstruction theorem needed on that extracted object class;
4. state the contradiction theorem that would exclude the minimal element.

Nothing from the frozen ledger counts as primary progress until this bridge is
written down exactly.

Hard stop:

- if no exact bridge can be stated without private shorthand or circular use of
  canonicity, the main route is blocked and the fallback route becomes primary.

### Phase 2: Try to prove the canonical extraction lemma

This is the main gate. It moves from the end of the draft to the front.

Required theorem target:

- a canonical critical-element extraction lemma, stable under compactness
  limits, harmless refinement, and representative relabeling.

Minimum acceptable success:

- an exact statement with a proof plan that genuinely controls all three
  invariance requirements on a standard critical-element sequence.

Hard stop:

- if after the first focused campaign the extraction lemma still depends on
  non-canonical window choice, finite-support truncation, or hand-frozen
  representative choices, stop the main route.

### Phase 3: Use template-defect and leakage only as obstruction tests on the canonical class

Only after Phase 2 succeeds:

- test whether the repaired template-defect theorem can be stated and proved on
  the extracted canonical class, not merely on a curated frozen witness family;
- test whether the repaired leakage firewall can be stated and proved in the
  same comparison currency against the hostile tiny-trigger family;
- combine them only if they feed a genuine `no-double-admissibility` theorem or
  an equivalent contradiction statement for the extracted minimal element.

What survives from the draft:

- the template-defect family theorem remains the first local test;
- the leakage firewall remains the second local test;
- both keep their kill rules.

What changes:

- they are no longer independent top-level routes;
- they are validation lemmas inside the critical-element bridge.

Hard stop:

- if these statements can only be proved on the frozen ledger and cannot be
  transferred to the extracted canonical class, they count as proxy identities
  only and do not justify continuing the main route.

### Phase 4: Assemble the contradiction or close the route

If Phases 0 through 3 all succeed, try to prove:

- no symmetry-normalized minimal blowup element in the chosen critical host
  space can satisfy the canonical extracted obstruction conditions.

That is the main-route endgame.

Hard stop:

- if the route yields only cleaner object definitions, exact sheet identities,
  or firewall language without a contradiction theorem on the PDE side, close
  the route as frontier mapping rather than prize-level progress.

## Fallback Route

### Direct geometric regularity theorem on accepted PDE objects

Run exactly one fallback theorem program:

- derive a recognized vorticity-direction regularity consequence from a
  quantified geometric hypothesis combining strain alignment and direction
  coherence in high-vorticity regions.

Target theorem shape:

- a condition stated directly on `u`, `omega`, `S`, and the vorticity direction
  field, which implies a Constantin-Fefferman-type regularity conclusion or a
  comparably accepted conditional regularity statement.

Immediate prerequisites:

- choose one theorem statement, not three directions;
- specify the downstream accepted criterion being verified;
- specify the exact place where the already-blocked `e_2`-alignment-alone route
  is being strengthened rather than repeated.

Hard stop:

- if the route collapses back to the known `s_2 > 0` obstruction,
- or merely reproduces Theorem C3-level conditionality without sharpening an
  accepted criterion,
- or depends on a broad exploratory search rather than one named lemma.

## Parked Routes

These are not active plan branches:

- recursive spill-versus-locking on shell-crossing exact triads,
- helical-sign packet functionals,
- broad non-Tao sidecar portfolio,
- any new packet-family work detached from the critical-element bridge.

They may supply mechanism evidence, but they do not get primary time unless the
main or fallback route produces a direct theorem need they uniquely satisfy.

## Resource Split And Time Gates

The program should stay narrow enough to kill itself honestly.

### First gate: short horizon

Within days to two weeks:

- finish Phase 0,
- write the exact bridge in Phase 1,
- decide whether the extraction lemma is even formally well-posed.

If that fails, switch to the fallback route.

### Second gate: medium horizon

Within one to two months:

- either produce real progress on the canonical extraction lemma,
- or declare the main route blocked.

Do not keep Routes 1 and 2 alive as standalone theorem projects if Phase 2 is
failing.

### Third gate: theorem horizon

Only after Phase 2 succeeds:

- spend major time on template-defect and leakage obstruction theorems.

If they do not feed a contradiction theorem on the extracted class, stop.

## What Counts As Real Progress

Only the following count as prize-facing progress:

1. a canonical extraction lemma on an accepted critical-element framework;
2. an obstruction theorem on that canonical class that contributes directly to a
   minimal-element contradiction;
3. a contradiction theorem excluding the critical element;
4. failing that, a new conditional regularity theorem stated purely in accepted
   PDE language.

The following do not count as primary progress:

- ledger consistency by itself,
- witness-specific frozen identities,
- new packet vocabulary,
- mechanism sketches without a PDE implication.
