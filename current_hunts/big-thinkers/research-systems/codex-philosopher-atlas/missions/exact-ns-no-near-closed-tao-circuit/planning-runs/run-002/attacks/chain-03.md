# Attack On Selected Chain 03

This file attacks `selected/chain-03.md`, which is internally labeled as
original Chain 04. The branch has one real strength: it tries to force the
best constructive-looking witness through honest exact bookkeeping instead of
living forever at dossier level. That is useful. But as written, the chain is
not clean. It quietly reintroduces discarded itinerary machinery, overstates
how much has already been earned from `F_SS`, and uses a very flexible
"minimal extension" clause that can turn a failed witness test into a rescue
exercise.

## Bottom Line

The branch is strongest as a bounded stress test of whether `F_SS(1/12)`
survives exact closure and dynamical scrutiny. It is weak as a standalone
mission branch because:

- it mixes the two promoted Step-6 gates with itinerary diagnostics that Step
  5 and Step 6 explicitly demoted or discarded;
- it assumes a small exact reduced system exists for the frozen witness, even
  though the recorded evidence is a Step-4 ledger, not an invariant finite-mode
  dynamical closure;
- it is less distinct than advertised from the joint-frontier branch, because
  inside the frozen one-bridge class Chain 03 already studies whether
  `F_SS`-type packets can survive both promoted screens;
- its repair stage is too permissive and too poorly calibrated to count as a
  hard falsification route.

## Step-Level Critique

### Step 1 - Freeze The Constructive Target And Honesty Rules

This step already cheats a little.

It says to freeze the scorecard as `G_tmpl`, `G_leak`, and the stage-order
diagnostics `t_clk`, `t_trig`, `t_rot`, `t_next` from the earlier itinerary
work. But the recorded Step-6 handoff did not promote those itinerary objects.
The repo already says:

- `Template-Defect Near-Closure` and `Windowed Spectator-Leakage Budget` are
  the only two promoted downstream objects;
- the itinerary fragments remain background only;
- the late transfer-start filter is not well-defined; and
- the early pre-trigger filter is not useful for the target theorem or
  counterexample question.

So Step 1 is not merely "freezing" a scorecard. It is manufacturing a stronger
compound target than the mission currently earns. That is a structural bias
toward the desired Tao-like narrative.

There is also a precision failure here: does "freeze `F_SS`" mean the full
family `F_SS(mu)` or the actual carried witness `F_SS(1/12)`? If `mu` stays
free, the branch is not frozen and the later "no broad sweep" language is
misleading. If the intent is the single witness `mu = 1/12`, it should say so.

### Step 2 - Derive The Exact Reduced Dynamics Of The Frozen Witness

This is the chain's biggest hidden assumption.

The local record gives `F_SS` as an exact Step-4 witness ledger: desired
channel forcing, spectator forcing, and an event trace on a normalized window.
That is not the same thing as an exact finite-dimensional reduced ODE that is
closed under all mandatory conjugates and closure-forced companions. Step 2
acts as if going from the dossier to a closed exact amplitude system is a
routine bookkeeping exercise. It is not. It may already be the core difficulty.

If honest bookkeeping forces a much larger packet, that is not some minor
technical snag. That is direct evidence that `F_SS` was only a favorable
screening witness, not a viable exact near-circuit packet. The kill condition
notes this possibility, but the whole chain still treats closure growth as an
edge case instead of one of the main expected obstruction modes.

This also exposes the evidentiary gap in the branch: nothing in the Step-4 or
Step-6 record proves that the packet sheet supporting the ledger is even close
to dynamically isolated under exact NS evolution.

### Step 3 - Test Whether The Frozen Witness Promotes Honestly

This step is better motivated than Step 1 or Step 2, but it is still sloppy in
its success criteria.

The good part is the insistence that static gate success is not enough. That is
true. If `F_SS` only passes `G_tmpl` and `G_leak` but has no credible staged
dynamics, it does not kill the firewall idea.

The problem is that the "true dynamical success" criteria are not actually
frozen in an honest way. "Showing the intended stage order strongly enough" is
not a mathematical condition. It invites post hoc interpretation, especially
because the branch has already imported `t_clk`, `t_trig`, `t_rot`, `t_next`
from an itinerary route whose late piece was rejected as not well-defined.

There is also a novelty problem. The local Step-6 record already treats
`F_SS(1/12)` as one of the families that passes both promoted screens. So what
Step 3 is really adding is not "promotion to near-circuit" from scratch. It is
the reintroduction of a dynamical itinerary burden that the main line of the
mission had already demoted. That makes the branch less like a new attack route
and more like a controlled attempt to resurrect a discarded criterion.

The kill condition against broad parameter sweeps is good in principle, but it
does not help if Step 1 never fixed `mu` exactly.

### Step 4 - Allow One Minimal Exact Extension If And Only If The Witness Nearly Works

This is the most dangerous step in the chain.

"Nearly works" is undefined. "One identified exact bottleneck" is undefined.
Those phrases create a large loophole for motivated patching.

Worse, the allowed repairs are not commensurate:

- one extra bridge;
- one extra packet companion; or
- one fixed shell-locked mode.

These are not variations of the same move. They change the packet class in
very different ways. An extra bridge in particular risks leaving the canonical
one-bridge class that Step 6 froze. If that happens, the branch is no longer
testing the same object class as the promoted gates, so "re-evaluate the same
scorecard" is not obviously legitimate.

The one-extension cap is also badly calibrated. If the branch fails after one
repair, that does not show the constructive route is false. It only shows this
very specific resource budget was exhausted. Conversely, if the extension does
work, the branch still owes an argument that the extension was not chosen by
post hoc tuning.

This step is exactly where a supposedly ruthless test can turn into a rescue
branch with just enough self-discipline language to sound honest.

### Step 5 - Package The Moonshot Verdict

The failure-atlas endpoint is genuinely good. If the branch ends with one named
collapse mode on honest exact bookkeeping, that is valuable.

But the packaging language still overreaches. Saying the verdict may apply to
"a slightly broader exact class genuinely earned in the branch" is too loose.
Nothing in Steps 1 to 4 earns broader-class conclusions unless the branch
changes class by extension, and if it does that, comparability to the frozen
Step-6 objects becomes weaker, not stronger.

The listed failure modes are also incomplete. The branch should admit at least
two additional decisive collapse types:

- the scorecard itself is illegitimate because it revives discarded itinerary
  content; and
- the repair exits the audited packet class, so the result no longer bears on
  the frozen one-bridge mission object in a clean way.

## Structural Weaknesses Of The Whole Chain

### 1. It revives a route the mission already demoted

The branch's distinctive ingredient is the stage-order story. But that is
exactly the part of the earlier itinerary route that Step 5 split and Step 6
declined to promote. As written, the chain is not just constructive. It is a
back-door reinstatement of a rejected third object.

### 2. It overstates how special `F_SS` is for the current mission

`F_SS(mu)` is the cleanest Step-4 friendly witness because it passed all three
Step-4 candidates on one exact sheet. That is real. But after Step 6, only two
objects remain promoted, and `F_SS(1/12)` is already part of the carried
friendly stress set for `G_leak` while both `F_SS(1/12)` and `F_SL(1/16)` are
known dossier-level passes for the two promoted screens. So the branch is not
starting from a uniquely mission-certified near-circuit seed. It is starting
from the most flattering old witness and asking for another chance under a new,
harder, partly unearned scorecard.

### 3. Its distinctness from Chain 03 is weaker than claimed

Inside the frozen canonical one-bridge class, Chain 03 already asks whether the
joint admissible region for `G_tmpl` and `G_leak` is empty, narrow, or has a
strong explicit survivor, and it places `F_SS`, `F_SL`, and `F_DT` on that
map. That already covers most of the honest content of a one-bridge witness
promotion.

This chain only becomes genuinely different in two ways:

- by adding the dynamical itinerary layer, which is exactly the unearned part;
  or
- by allowing class-changing extensions, which weakens comparability to the
  frozen Step-6 object set.

So its advertised "meaningfully different" status is only partially true.

### 4. The kill conditions are asymmetrically calibrated

The branch is strict about early bookkeeping honesty but loose about late-stage
repair discretion. That asymmetry favors survival of the constructive branch.
It is easier to smuggle optimism through "near miss" and "minimal extension"
language than through the initial freeze language.

If the goal is ruthless truth-finding, the branch should be stricter exactly
where motivated rescue is most tempting, not exactly where bookkeeping is
easiest.

### 5. It conflates three evidentiary layers

The chain slides across:

- Step-4 dossier pass;
- Step-6 promoted static gates; and
- exact reduced dynamical near-circuit behavior.

Those are not the same kind of evidence, and the branch never states the
transfer principle connecting them. Without that bridge, "promote the witness"
is rhetoric, not a logically controlled program.

### 6. Even its bookkeeping has a small warning sign

The selected file is `selected/chain-03.md` but the copied text is labeled
"Chain 04." That is minor compared to the mathematical issues, but it is still
a bad sign in a branch whose whole point is to be exact about what was frozen,
what was promoted, and what was discarded.

## Fair Assessment

What is genuinely strong here:

- starting from a named positive witness instead of inventing a new family is
  disciplined;
- insisting on honest exact bookkeeping before claiming anything constructive
  is the right instinct;
- the failure-atlas endpoint could produce useful negative knowledge.

What is not strong:

- the branch does not respect the Step-6 demotion of itinerary content;
- it assumes reduced exact dynamics are accessible before proving the witness is
  dynamically isolable; and
- its extension step is too flexible to count as a clean falsification route.

Best fair verdict: keep the branch only as a tightly bounded stress test of the
single packet `F_SS(1/12)`, with no revived itinerary scorecard and no
class-changing repair. As written, it is too biased, too redundant, and too
loose at the exact point where honesty matters most.
