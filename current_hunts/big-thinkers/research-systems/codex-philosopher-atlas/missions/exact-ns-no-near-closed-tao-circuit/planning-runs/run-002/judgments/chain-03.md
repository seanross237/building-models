# Judgment for Chain 03

## Overall assessment

Chain 03 should survive judgment, but only in a much narrower form than the
selected version. The attacker is right that the original branch quietly
reintroduces demoted itinerary material, assumes exact reduced dynamics too
easily, and leaves a large late-stage rescue loophole. The attacker overstates
only one point: a disciplined exact audit of one named constructive witness is
still useful and is not fully redundant with the joint-frontier branch.

The right refinement is to keep the branch as a bounded exact stress test of
`F_SS(1/12)`, using only the promoted Step-6 gates
`G_tmpl`
and
`G_leak`
as pass/fail criteria. Stage-order data may still be recorded, but only as
secondary diagnostics. The "minimal extension" repair step should be removed.

## Major critique judgments

### 1. Step 1 revives demoted itinerary content and does not freeze the witness sharply enough

**Verdict: Valid**

This is the strongest early critique.

- Step 5 and Step 6 explicitly keep
  `Template-Defect Near-Closure`
  and
  `Windowed Spectator-Leakage Budget`
  while splitting and then discarding the itinerary route as a promoted object.
- The original Step 1 does more than "freeze" the scorecard. It promotes
  `t_clk`,
  `t_trig`,
  `t_rot`,
  and
  `t_next`
  back into the branch's success language.
- The original chain also says
  `F_SS(mu)`
  when the honest frozen witness for this branch should be the carried point
  `F_SS(1/12)`.

Refinement:

- freeze the single witness
  `F_SS(1/12)`;
- freeze only
  `G_tmpl`
  and
  `G_leak`
  as pass/fail gates;
- keep stage-order/timing information only as descriptive output.

### 2. Step 2 assumes an exact closed reduced system before earning it

**Verdict: Valid**

The local record supports dossier-level and gate-level witness passes. It does
not already support the existence of a small exact invariant ODE for
`F_SS(1/12)`.

The original chain does mention packet growth as a kill condition, but it
still treats honest reduction as the default flow instead of one of the main
obstruction tests. For this branch, exact closure is not bookkeeping around the
real task. It is the real task.

Refinement:

- move exact closure and class retention to the center of the branch;
- allow the chain to end immediately with a constructive non-isolability memo
  if the witness exits the audited class or requires arbitrary truncation.

### 3. Step 3 uses vague dynamical success language and overstates novelty

**Verdict: Partially valid**

The critique is right that "showing the intended stage order strongly enough"
is too loose to serve as a promoted success criterion. It is also right that
`F_SS(1/12)` already appears in the Step-6 record as part of the friendly
stress set for
`G_leak`,
and that both
`F_SS(1/12)`
and
`F_SL(1/16)`
already pass the two promoted static screens.

But the novelty objection is overstated if the branch is rewritten correctly.
A witness-local exact closure-and-dynamics audit is still different from the
joint-frontier chain. The distinct work here is not "promote a static witness
again." It is "test whether the best constructive-looking witness survives
honest exact dynamics at all."

Refinement:

- dynamic success is defined first by the promoted Step-6 gates;
- stage-order data is secondary descriptive evidence only;
- the branch must present itself as a witness audit, not as a reinstated
  itinerary program.

### 4. Step 4 turns the branch into a rescue exercise

**Verdict: Valid**

"Nearly works" and "one identified exact bottleneck" are undefined, and the
allowed repairs are not commensurate. An extra bridge, an extra companion, and
a shell-locked mode do not preserve comparability in the same way, and an
extra bridge would leave the frozen one-bridge class outright.

This is the point where the original chain is most asymmetrically calibrated:
strict on early bookkeeping, permissive exactly where motivated rescue becomes
tempting.

Refinement:

- remove the discretionary repair step entirely;
- any added mode must be either already forced by exact closure, in which case
  it belongs to Step 2 bookkeeping, or else it is disallowed;
- no class-changing extension counts as a legitimate rescue.

### 5. Step 5 packages conclusions more broadly than the evidence earns

**Verdict: Valid**

The constructive failure-atlas endpoint is genuinely good and should stay. The
overreach lies in the original language about conclusions for "a slightly
broader exact class" and in treating the branch as if it could cleanly bear on
the firewall idea after witness-local work alone.

The attacker is also right that two additional failure modes should be named
explicitly:

- the branch used an illegitimate scorecard by reviving demoted itinerary
  content;
- the witness exited the audited packet class under honest closure.

Refinement:

- scope every verdict to the frozen witness unless a broader statement is
  actually proved;
- treat class exit and scorecard illegitimacy as explicit constructive failure
  endpoints.

### 6. The whole chain overstates how special `F_SS` is and how different it is from Chain 01

**Verdict: Partially valid**

This criticism is right in mission-level terms but too strong in constructive
terms.

What is valid:

- after Step 6,
  `F_SS(1/12)`
  is not unique as a pass for the promoted gate pair, because
  `F_SL(1/16)`
  also passes both promoted screens;
- the selected branch was less distinct than advertised once it mixed in the
  same promoted gates already studied in the joint-frontier chain;
- the original chain slid too quickly across three evidentiary layers:
  Step-4 dossier evidence,
  Step-6 promoted static gates,
  and exact dynamical near-circuit behavior.

What is overstated:

- `F_SS(1/12)` is still a reasonable constructive seed because it is the
  cleanest recorded positive witness with the strongest late-stage dossier
  profile on disk;
- a witness-local exact stress test remains meaningfully different from a
  class-wide frontier map if it avoids revived itinerary scoring and avoids
  class-changing repair.

Refinement:

- state clearly that the branch is about one frozen witness, not the full
  one-bridge class;
- separate static-gate evidence from dynamical evidence explicitly at every
  stage.

## Refined judgment

Keep Chain 03 only in this narrowed form:

- freeze the single witness
  `F_SS(1/12)`;
- use only the promoted Step-6 gates
  `G_tmpl`
  and
  `G_leak`
  as hard pass/fail criteria;
- make exact closure the first real test;
- forbid discretionary repair or class-changing extension;
- package the outcome symmetrically as either witness-local anti-obstruction
  evidence or a witness-local constructive failure atlas.

Under those revisions, the branch remains worth keeping because it preserves
the portfolio's highest-upside constructive stress test while no longer
smuggling in discarded itinerary content or rescue-by-patching.

## Probability assessment

**Probability that the refined chain yields a presentable result: 0.74**

Reason for this level:

- the branch is now tightly bounded around one named witness and one frozen
  scorecard, which makes presentable failure much more likely;
- exact closure failure by itself would already be a clean constructive
  negative memo;
- an honest dynamic survival memo is still possible but clearly lower
  probability than a failure atlas;
- a true presentable near-circuit challenger remains possible but is the least
  likely endpoint.
