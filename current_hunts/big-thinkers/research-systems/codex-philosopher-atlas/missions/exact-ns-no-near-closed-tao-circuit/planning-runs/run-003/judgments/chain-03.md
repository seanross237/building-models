# Judgment for Chain 03

Note: the selected source file is `selected/chain-03.md`, but its internal
heading says `Chain 01`. This judgment follows the selected file path used by
the run.

## Overall assessment

Chain 03 should survive judgment, but only in a materially narrower form. The
attacker is right about the branch's main miscalibration: on the current
record, this is not primarily a fresh five-step family theorem program. It is
first a provenance test of whether the Step-4 `F_SL(rho)` draft arithmetic can
be promoted into repaired same-currency authority on the Step-5 / Step-6
defect sheet.

If that promotion succeeds, the likely mathematical endpoint is modest:
a carried-subfamily admissibility result on
`0 < rho <= 1/16`
for repaired `G_tmpl`,
with the boundary likely collapsing to the already-carried witness
`rho = 1/16`.
If that promotion fails, the honest endpoint is not "keep deriving." It is a
source-basis failure memo.

The refined chain should therefore:

- cite the Step-6 freeze rather than re-found the branch from scratch;
- fix scope to the carried Step-6 range unless wider reopening is declared as
  new scope;
- treat the `rho`-ledger step as an adversarial promotion audit, not routine
  derivation;
- remove the unrepaired `(-3 + sqrt(10)) / 2` endpoint from the live route;
- front-load the no-closure / no-transfer warning instead of leaving it to a
  late packaging step.

## Major critique judgments

### 1. Step 1 duplicates the Step-6 freeze, risks stale arithmetic, and leaves scope ambiguous

**Verdict: Valid**

This critique lands.

- Step 6 already froze the live promoted object:
  repaired `G_tmpl`,
  thresholds `(1/4, 49/256)`,
  fixed window `I = [0, 1]`,
  and the carried family question on
  `0 < rho <= 1/16`.
- The selected chain's language about a "larger possible endpoint" is unsafe in
  this repaired branch. The Step-4 draft root
  `(-3 + sqrt(10)) / 2`
  belongs to the old `lambda_spec = 1/4` sheet, not the repaired
  `49/256` sheet.
- The selected chain also leaves open whether "admissible interval" means the
  original Step-4 family
  `0 < rho <= 1/12`
  or the carried Step-6 subfamily
  `0 < rho <= 1/16`.
  That ambiguity matters.
- The attacker's point about the kill condition is also right:
  the repository can already name the family `F_SL(rho)`;
  what is still missing is promoted repaired-ledger authority for the exact
  `rho`-dependent defect formulas.

Fair caveat:
a short Step-1 authority memo is still useful, but only if it cites Step 6 and
Step 8 as already-frozen authority instead of pretending to refound them.

### 2. Step 2 is the real crux and must be treated as a provenance audit

**Verdict: Valid**

This is the core criticism, and the repository record supports it directly.

- The Step-6 next-test note explicitly says the local record still needs one
  explicit `rho`-dependent formula ledger for
  `Delta_tmpl`
  and
  `Delta_spec`.
- The Step-4 pro-side report explicitly says the concrete pro-family material on
  disk was only family-level `O(mu)` / `O(rho)` descriptions plus a stalled
  draft in a runtime log, later rechecked with
  `code/pro_circuit_dossier_check.py`.
- That script is useful evidence, but it is not automatically theorem-grade
  repaired authority.
- The attacker's warning about permissive wording like "use existing checked
  arithmetic or scripts" is therefore correct. This branch should not allow a
  hand-entered checker to silently become proof authority.

One caveat is overstated by the attacker:
the branch does not need a philosophical explanation of why
`Delta_tmpl(rho)` might be constant before it can use a formula.
It needs a same-currency derivation or citation path, not an interpretive
essay. But that does not weaken the main critique.

### 3. Step 3 is either algebra or blocked, and the old `sqrt(10)` root should be removed

**Verdict: Valid**

The attacker's logic is right here.

- On the draft formulas already on disk,
  `Delta_tmpl = 1/4`
  and
  `Delta_spec = 3 rho + rho^2`,
  the repaired carried-range computation is short:
  `Delta_tmpl <= 1/4`
  is saturated identically,
  and
  `3 rho + rho^2 <= 49/256`
  hits equality exactly at
  `rho = 1/16`.
- If those formulas are not promoted honestly, then Step 3 has no exact ledger
  to solve.
- The selected chain's instruction to keep testing
  `(-3 + sqrt(10)) / 2`
  is the clearest stale-arithmetic leak in the branch.
- The phrase "true admissible interval" also overreaches unless the chain first
  declares that it is reopening the old Step-4 family beyond the carried
  Step-6 range.

So Step 3 should be rewritten as a carried-range boundary and sharpness audit,
not a generic frontier analysis.

### 4. The no-overclaim guardrail is necessary but belongs earlier

**Verdict: Valid**

The attacker is right that this discipline should be front-loaded.

- Step 8 already established that admissibility-style frozen-ledger data does
  not by itself yield honest finite closure for the carried witness
  `F_SS(1/12)`.
- That is not a theorem about `F_SL(rho)`, so the comparison does not create new
  family math.
  But it is strong enough to freeze a general warning:
  template admissibility is not closure.
- The selected chain's question about whether "one extra exact hypothesis"
  might turn the family theorem into something closure-relevant is too loose.
  In practice that invites reopening other branches under softer language.

The refined version should keep a scope memo, but treat it as a premise and a
packaging constraint, not as late-stage discovery.

### 5. Step 5 risks overpackaging a cleanup as a theorem

**Verdict: Partially valid**

The attacker is right about the danger, but slightly too strong about the
selected chain's intent.

- The selected chain does at least allow modest endpoints:
  a witness-only memo or a bounded negative, not only a full theorem.
- But the branch is still framed as the "cleanest surviving positive theorem
  route," and that makes theorem inflation a real packaging risk.
- The critique is especially persuasive that the branch should treat
  source-basis failure as a first-class successful outcome, not as an awkward
  fallback after a theorem-shaped arc collapses.
- The attacker is also right that the final safeguards must block a specific
  overclaim:
  repackaging old Step-4 draft arithmetic as if it were newly earned repaired
  theorem data.

So the selected Step 5 is not invalid, but it needs stricter burden control and
more neutral endpoint language.

### 6. The real bottleneck is provenance, not algebra

**Verdict: Valid**

This is the best structural summary in the attack.

The repository already suggests a family ledger and even the likely repaired
endpoint. What Step 6 withholds is promotion. That means the branch's first
serious question is source-basis legitimacy, not symbolic difficulty.

### 7. The branch risks mixing repaired and unrepaired arithmetic

**Verdict: Valid**

### 8. The middle of the chain is either trivial or blocked, and the branch is closer to the Step-6 handoff than advertised

**Verdict: Partially valid**

This is directionally right, but overstated if taken literally.

- The attacker is right that once the formulas are promoted, the mathematical
  boundary computation is small, and if they are not promoted, the branch
  should fail fast.
- The attacker is also right that Step 6 already names most of the live issues:
  the carried range,
  the missing `rho`-ledger,
  boundary sharpness,
  and the need for explicit transfer scope.

But the chain is not wholly redundant.
There is still planning value in sequencing:

- the authority freeze,
- the adversarial promotion audit,
- the carried-range boundary proof if promotion succeeds,
- and the no-overclaim packaging.

So the refined chain should be much smaller than the selected one, but not
discarded as empty.

## Refined judgment

Keep Chain 03 only in this narrower form:

- freeze Step-6 authority instead of re-freezing new foundations;
- fix scope to repaired `G_tmpl` on the carried family domain
  `0 < rho <= 1/16`;
- treat the Step-4 `F_SL(rho)` formulas as draft candidates that must survive an
  explicit repaired-ledger promotion audit;
- remove the old `sqrt(10)` endpoint from the live computation path;
- front-load the warning that any success is only family-local template
  admissibility unless an additional same-currency transfer theorem is actually
  proved;
- allow the branch to end cleanly with either a carried-range admissibility
  memo, a witness-only memo at `rho = 1/16`, or a source-basis failure memo.

Under those revisions, the branch remains worth keeping.
Its best honest output is probably not a broad theorem.
It is a disciplined answer to one narrower question:
can the repository upgrade the suggested `F_SL(rho)` arithmetic into repaired
theorem-grade data, and if so does the carried boundary really sit at
`rho = 1/16`?
