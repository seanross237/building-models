# Chain 02 - Admissibility Versus Exact-Closure Separation

## Central Premise

Step `008` may have exposed the real mission bottleneck.
The current record already contains packets that pass repaired static
admissibility screens but still do not yield one finite honest exact closed
subsystem.

The branch should therefore test, on the frozen canonical one-bridge ledger,
whether repaired `G_tmpl`, repaired `G_leak`, or their conjunction implies any
honest finite-closure statement at all, or whether exact closure is a genuinely
separate theorem object.

## Verifiable Result Target

Either:

- a clean separation proposition showing that one or both Step-6 promoted gates
  do **not** imply honest finite exact closure on the frozen ledger;
- a corrected implication memo showing that a stronger but still exact closure
  predicate follows once one extra hypothesis is added explicitly; or
- a bounded negative showing that the repository still cannot state finite
  closure in a non-arbitrary way on the same packet semantics.

## Why This Chain Is Meaningfully Different

This chain is relation-facing rather than family-facing or search-facing.
It studies the implication map between existing promoted objects and honest
closure.
It does not seek a new packet first, and it does not attempt a broad support
theorem over all realizations.

## Ordered Steps

### Step 1 - Freeze The Implication Sheet To Be Tested

Depends on: none.

Action: freeze repaired `G_tmpl`, repaired `G_leak`, the Step-8 verdict on
`F_SS(1/12)`, and the fixed same-currency packet ledger from Steps `006`-`008`.

Action: write the exact implication questions explicitly:
does `G_tmpl` imply finite closure,
does `G_leak` imply finite closure,
and does joint pass of both gates imply finite closure on the frozen ledger?

Expected output: one implication sheet that names the three exact relations the
branch will test.

Kill condition: if the branch cannot state the implication targets more
precisely than a slogan like "small leakage should help closure," stop.

### Step 2 - State One Honest Exact-Closure Predicate On The Same Ledger

Depends on: Step 1.

Action: define one finite-closure predicate in the same packet language, such
as existence of one finite exact closed subsystem with no arbitrary truncation
boundary beyond the forced canonical bookkeeping.

Action: keep this predicate strictly inside the frozen packet semantics and
same-currency rules used in Step `008`.

Expected output: one exact closure predicate memo, suitable for implication and
separation statements.

Kill condition: if finite closure can only be formulated by changing support
semantics, inserting an unearned truncation rule, or switching burden currency,
stop and report that exact closure is not yet an honest predicate here.

### Step 3 - Populate The Relation Matrix With The Carried Witnesses

Depends on: Step 2.

Action: place
`F_SS(1/12)`,
`F_SL(1/16)`,
and
`F_DT(delta, eta)`
on the relation matrix:
which promoted gates each passes or fails, and what is actually known about
finite closure for each.

Action: use `F_SS(1/12)` as the primary Step-8 separation witness if the same
ledger really supports that reading.

Expected output: one witness matrix for
`G_tmpl`,
`G_leak`,
joint pass,
and finite-closure status.

Kill condition: if the putative separation witness changes packet semantics,
scorecard, or support conventions relative to the promoted objects, stop.

### Step 4 - Prove The Strongest Honest Separation Or Minimal Repair

Depends on: Step 3.

Action: prove the strongest relation actually earned on the frozen ledger:
for example,
joint repaired admissibility does not imply finite honest closure,
or repaired leakage does not imply isolability without one extra explicit
support-level hypothesis.

Action: if pure separation is too strong, identify the minimal added exact
hypothesis that converts the failed implication into a true statement.

Expected output: one proposition candidate or scope-correction memo describing
exactly how admissibility and closure are related.

Kill condition: if the branch can conclude only that the objects are
"morally related" without one precise separation witness or one precise repair
hypothesis, stop.

### Step 5 - Decide The Downstream Consequence For Mission Objects

Depends on: Step 4.

Action: state whether the mission should keep repaired `G_tmpl` and repaired
`G_leak` as screening tools only, or whether one third closure-level object now
needs to be promoted explicitly.

Expected output: one downstream-object memo that either freezes a relation
result or names the exact missing closure object now required.

Kill condition: if the final memo reopens broad definition work without first
using the Step-8 evidence to state one exact relation failure, the branch
failed.
