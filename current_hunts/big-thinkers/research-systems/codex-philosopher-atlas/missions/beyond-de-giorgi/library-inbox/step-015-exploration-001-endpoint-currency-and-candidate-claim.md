## Findings

### 1. Endpoint family

- [VERIFIED] The only explicit geometry-facing theorem endpoint family named on
  disk is the Constantin-Fefferman-type direction-regularity route.
  `MISSION.md` names Constantin-Fefferman directly as the repository's concrete
  geometry criterion family.
- [VERIFIED] The broader framework freeze says the only locally frozen theorem
  floor is the suitable-weak / LEI CKN/Lin endpoint-exclusion package, but the
  current geometry chain asks Step 1 to choose between a
  Constantin-Fefferman-type criterion and a named full-stretching contradiction
  target.
- [INFERRED] The local record supports the criterion family more concretely
  than a package-free full-stretching contradiction target.
  Full stretching is the mechanism screen the branch must eventually touch, but
  the repository does not freeze a separate theorem-level contradiction target
  for it before package choice.

### 2. Gain currency

- [INFERRED] The correct same-currency unit is `criterion input`.
- [INFERRED] Reason:
  once the endpoint family is frozen to a Constantin-Fefferman-type criterion,
  the only honest gain that counts at Step 1 is a reduction in the criterion
  input burden, not prettier stretching language in another ledger.
- [VERIFIED] The local geometry screen already forbids claims that only make a
  local geometric proxy look cleaner without changing a full-stretching term or
  a theorem-facing endpoint input.

### 3. Minimum non-cosmetic gain in that currency

- [INFERRED] Minimum non-cosmetic gain:
  one NS-specific insertion that is strictly cheaper than importing the
  comparator's criterion-scale direction-regularity burden and that still
  points to a smaller full-stretching contribution rather than to descriptive
  geometry alone.
- [INFERRED] Concretely, the step should not count
  `direction coherence looks favorable on intense regions`
  as gain.
  It would count only if the new ingredient reduced the amount of
  criterion-scale direction control one must assume relative to the
  Constantin-Fefferman comparator.

### 4. Candidate theorem-facing claim

- [PROPOSED] The only package-free candidate insertion line the local record
  still permits is:

  If a Navier-Stokes-specific dynamic coherence-persistence condition on
  intense-vorticity regions yields a criterion-scale direction-control input
  that is strictly cheaper than the static Constantin-Fefferman comparator and
  the gain is witnessed through a smaller full-stretching contribution rather
  than through package adaptation, then that input should insert into a
  Constantin-Fefferman-type regularity criterion.

- [INFERRED] This candidate is narrower than a generic
  `geometry helps stretching`
  slogan, but it is still only an insertion line.
  It does not yet specify a package, observable class, or proof mechanism.
- [VERIFIED] The only local family that made such an insertion line even
  conceptually live was the hybrid
  `direction coherence + tube persistence`.
- [VERIFIED] The same local record also says that this hybrid is only
  `informative but dynamically weak` on the audited package, so the candidate
  claim remains provisional rather than earned.

### 5. What endpoint-level success would mean

- [INFERRED] Success at the endpoint level would mean more than surviving a Tao
  screen or identifying attractive geometry.
- [INFERRED] It would mean that a new NS-specific dynamic input actually
  changes the criterion ledger:
  it replaces or weakens the comparator's direction-regularity burden in a way
  that still talks to full stretching.

## Conclusion

- Endpoint family frozen:
  `Constantin-Fefferman-type direction-regularity criterion`
- Gain currency frozen:
  `criterion input`
- Minimum non-cosmetic gain:
  a strictly cheaper criterion-scale insertion than the static comparator's
  direction-control burden, still certified by a full-stretching-facing gain
- Candidate claim status:
  `[PROPOSED]` only.
  The local record permits a narrow hybrid insertion line, but not a fully
  earned package-free theorem statement.
