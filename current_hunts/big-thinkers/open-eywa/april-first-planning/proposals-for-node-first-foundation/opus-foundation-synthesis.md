# Foundation Synthesis: What To Actually Build

## The Field

Eight proposals. They cluster into three tiers:

**Tier 1 — Codeable.** These two are concrete enough that an engineer could start building from them:

- **furnace-node-first.md** — 5 statuses, 3 outcomes, per-run directories, three handoff validation functions with actual Python code, cost.json, concrete dataclasses. Derives sub-phases from file inspection. Aligned tightly with the debate conclusions.
- **opus-node-first-core.md** — 5 statuses, explicit phase tracking, contract/human split (JSON for orchestrator, markdown for humans), `tick(tree) -> list[Action]` pure function design, AgentInput/AgentOutput protocol, schema versioning.

**Tier 2 — Conceptually strong but abstract.** Good ideas, but you'd still need to design the implementation:

- **operator-trust-node-first.md** — Good trust framing. 9 statuses (too many). Clear on what completion means.
- **codex-balanced-node-first.md** — Most balanced coverage of all topics. Clean writing. Still abstract.
- **reliability-hard-guarantees.md** — Strongest on enforcement. Good list of hard rules. No code.

**Tier 3 — Single-lens.** Valuable perspectives but too narrow to build from alone:

- **extensibility-experimentability.md** — Extensibility and A/B testing lens. Good ideas, light on mechanism.
- **measurement-and-ablation.md** — Pure measurement lens. Shortest.
- **operator-legibility.md** — Human readability lens.

## Verdict

**Furnace is the strongest foundation. Opus has the strongest architectural idea.**

Furnace wins because it's the most directly implementable, stays closest to the debate conclusions, and avoids overcomplication. Its three validation functions, per-run directories, and cost cap enforcement are exactly the kind of concrete moves that prevent the "great doc, unclear implementation" failure mode.

Opus's `tick(tree) -> list[Action]` pure function design is the single best idea across all proposals. It makes the orchestrator trivially testable without a filesystem, without processes, without LLM calls. That idea should be adopted wholesale.

Opus's contract/human split (JSON for machine, markdown for humans) is elegant but adds dual-write overhead. Agents would need to produce structured JSON AND readable markdown, or we'd need an auto-rendering step. That's complexity that doesn't earn its keep yet. The simpler path: keep files as markdown, have the orchestrator check file existence and read specific fields where needed (which is what Furnace's three validation functions already do). We can add structured contract files later if markdown parsing becomes a real problem.

Opus's explicit phase tracking is a genuinely useful addition to Furnace's model. Furnace derives sub-phases from file inspection, which is clean but makes the current phase invisible to humans reading the node directory. An explicit phase field (stored alongside status, written only by the orchestrator) gives you the inspectability benefit without bloating the status enum.

## The Recommended Design

Take Furnace as the base. Add three things from Opus. Drop one thing from Furnace.

### Status Model

5 statuses. Closed enum. This never changes.

```
pending -> running -> done
                   -> failed
                   -> timed_out
```

Terminal outcomes (separate field, stored alongside status):

| Outcome | Meaning | Required artifact |
|---|---|---|
| completed | Produced the required output | final-output.md |
| escalated | Explicitly punted to parent | escalation.md |
| cancelled | Parent killed this node | none |

Failure reasons (separate field, stored alongside status for failed/timed_out):

| Reason | Trigger |
|---|---|
| cost_cap_exceeded | total_usd > cap_usd |
| max_retries_exceeded | run_count >= limit without satisfaction |
| zombie_detected | PID dead, no output |
| api_timeout | LLM call or process exceeded wall-clock limit |

### Phase (from Opus, added to Furnace)

When a node is `running`, the orchestrator writes an explicit phase:

```
retrieving | planning | reviewing | deciding | executing | evaluating | synthesizing | working | waiting_compute
```

Phase is written to the status file by the orchestrator, not by agents. It is informational — the state machine only cares about the 5 statuses. But it makes the node inspectable at a glance: you can see "this node is running, and it's currently in the reviewing phase" without having to deduce it from which files exist.

New phases can be added without changing the state machine.

### Node Directory Layout

Furnace's layout, with shortened filenames:

```
<node>/
  input/
    goal.md
    parent-instructions.md
    context.md                  # accumulated context from prior steps
    knowledge.md                # library retrieval
    instructions-<role>.md      # generated by orchestrator before each spawn
  output/
    plan.md
    review.md
    final-output.md
    state.md
    escalation.md
  for-orchestrator/
    status                      # "running"
    phase                       # "planning" (NEW: from Opus)
    outcome                     # "completed" (only when terminal)
    mode                        # "planner"
    eval-decision               # "continue" | "replan" | "escalate"
    cost.json
    waiting                     # background computation marker
    computation-result          # background computation done marker
  system/
    runs/
      run-001/
        meta.json               # pid, role, model, variant, started_at, finished_at, cost, tokens, exit_reason, artifacts_produced
        loop.jsonl
        usage.json
        stdout.log
        stderr.log
    artifacts/
    jobs/
  children/
    step-01-<slug>/
    step-02-<slug>/
```

### The Orchestrator As Pure Function (from Opus)

This is the key architectural addition. The orchestrator's core loop becomes:

```python
def tick(tree: dict[str, NodeView]) -> list[Action]:
    """Read the whole tree. Return actions. No side effects."""
    actions = []
    for node_id, node in tree.items():
        if node.status == "pending":
            actions.append(SpawnAgent(node_id, role="librarian"))
            actions.append(SetPhase(node_id, phase="retrieving"))
            actions.append(SetStatus(node_id, new="running"))
        elif node.status == "running":
            actions.extend(tick_running(node))
        # terminal: no action
    return actions
```

Actions are value objects. An executor applies them to the filesystem. The `tick()` function is trivially testable: construct a tree in memory, call tick(), assert on the returned actions. No filesystem, no processes, no LLM calls. This is how you build the offline test suite cheaply.

### Three Handoff Validations (from Furnace)

Keep Furnace's three validation functions exactly as specified:

1. **Node creation handoff** — goal exists, status file exists, status is pending
2. **Run completion handoff** — role-specific required artifacts exist (or escalation.md exists for escalation-eligible roles)
3. **Node completion handoff** — before writing `done`, validate the terminal artifact matches the outcome

These are called by the action executor, not by tick(). Tick decides what should happen; the executor validates and applies.

### Cost Cap Enforcement (from Furnace)

Furnace's cost.json and check_cost_cap() exactly as specified. Hard kill, no model consultation.

### Per-Run Directories (from Furnace)

Each agent run gets its own directory under `system/runs/run-NNN/`. Run records include a `variant` field for future A/B testing (from Opus). Monotonic numbering. Never overwritten.

### What I'd Drop

**Opus's contract/human split.** Not yet. The overhead of dual-writing JSON + markdown doesn't pay for itself at this stage. If we later find the orchestrator struggling to parse markdown, we can introduce structured contract files per-role. But right now, the three validation functions (which check file existence, not content) are sufficient.

**Opus's schema versioning.** Premature. We don't have stable schemas yet. Version them when we have a second version.

## Key Decisions Explained

**Why 5 statuses and not the v1's 12+?**
The v1 status enum encoded workflow sub-phases as distinct statuses. That means every new workflow variation risks adding a new status. With 5 statuses + phase, the state machine is closed forever. New workflow steps add phases, not statuses. The orchestrator's transition logic stays the same size.

**Why explicit phase instead of pure file-inspection?**
Furnace's file-inspection approach is elegant but makes the current phase invisible to anyone reading the directory. An explicit phase file costs almost nothing and dramatically improves inspectability. The tradeoff is one more file the orchestrator writes — trivial.

**Why tick() as a pure function?**
This is the difference between "we have offline tests" and "offline tests are easy to write." If tick() is pure, every test is: build tree data, call tick(), assert actions. No mocking, no filesystem setup, no process management. That's the kind of testing foundation that makes it safe to iterate fast.

**Why keep markdown files instead of JSON contracts?**
The orchestrator needs to know three things: does the file exist? Is it the right file for this outcome? Is the cost under cap? None of those require parsing the file's content structure. The three validation functions handle this. Adding a parallel JSON layer doubles the work for a problem we don't currently have.

**Why per-run directories?**
The v1 system overwrites active-run.json on each spawn. If a node gets three runs, the history of the first two is partially lost. Per-run directories make every run independently inspectable, and they're the natural unit for A/B comparison.

## Biggest Remaining Risks

1. **Orchestrator conditionals.** The 5-status model pushes sub-phase logic into `tick_running()`, which could become a pile of conditionals. Mitigation: structure it as a decision table (status, phase, file conditions) -> action. Test each row.

2. **Phase/file disagreement.** The orchestrator writes the phase, but the phase should be derivable from what files exist. If they disagree, the system is confused. Mitigation: the phase is always set by the same code that triggers the action. And phase is a hint, not a gate — the validation functions check files, not phase.

3. **File race conditions.** The orchestrator polls, agents write files. If the orchestrator reads between an agent writing plan.md and finishing its run, the phase could be wrong. Mitigation: agents write artifacts before signaling completion (they never write status or phase). The polling interval provides natural debounce.

4. **"Just one more file" creep.** Pressure to add more well-known files to the node contract. Mitigation: any new protocol file must be added to the validation functions and this doc. If it's not worth validating, it's not worth canonizing.

## Implementation Order

1. Define `NodeStatus` (5 values) and `NodeOutcome` (3 values) enums
2. Define the Action types (SpawnAgent, CreateChild, SetStatus, SetPhase, etc.)
3. Implement `tick()` as a pure function with a decision table
4. Implement the three handoff validation functions
5. Implement the action executor (applies actions to filesystem)
6. Set up the per-run directory structure under `system/runs/`
7. Implement `cost.json` tracking and cap enforcement
8. Write offline tests for every row of the decision table
9. Migrate or clean-break from v1
