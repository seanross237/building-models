## Frozen Commitments You Must Inherit

All candidate notions in this step must inherit the Step-1 freezes:

- finite role-labeled helical packets are the primary object;
- real-valuedness is handled by mandatory conjugate completion in one fixed
  canonical representation;
- Tao-likeness must preserve stage order, delayed-threshold logic,
  tiny-trigger centrality, amplitude hierarchy, and time-scale separation;
- every candidate must declare its helical sign sheet, amplitude
  normalization, and phase anchor or equivalent canonical gauge;
- every candidate must use predeclared leakage or behavioral criteria on one
  fixed finite window;
- every candidate must name exactly one downstream gate;
- later negative conclusions are allowed to land only in:
  `not well-defined`,
  `not robust after canonicalization`,
  or `not useful for the target theorem or counterexample question`.

No candidate may evade these freezes by changing packet semantics,
redefining support by convenience, or importing Tao's averaged construction by
hand.

## Required Deliverables

Produce all of the following inside `RESULTS.md`:

1. **Candidate table**
   - list each candidate notion, with at most four total;
   - give each candidate a short name and one-sentence intent;
   - classify each as:
     - `obstruction-oriented`
     - `construction-oriented`
     - or `dual-use`.

2. **Definition sheet for each candidate**
   - state the exact criterion, formula, or sharply finite checklist;
   - state the packet class it lives on;
   - state the exact role of leakage or behavioral thresholds;
   - state the finite time or activation window it uses;
   - state the declared phase/helical-sign/canonicalization inputs it depends
     on.

3. **Exact interaction template sheet**
   - for each candidate, write the exact Fourier/helical interaction template
     it is meant to track;
   - name which couplings are desired, which extra couplings are automatically
     forced, and which interaction data later steps will still need;
   - keep this concrete enough that Step 3 can audit robustness and Step 4 can
     prepare normalized packet tests.

4. **Downstream-gate ledger**
   - for each candidate, name exactly one downstream gate:
     one invariant dynamical observable on one stated packet family and finite
     window, or one invariant admissibility sheet for adversarial exact search;
   - explain briefly why that gate is the right next audit for that candidate.

5. **Candidate verdict**
   - state whether Chain Step 3 is now well-posed;
   - if yes, identify which candidates survive into robustness audit and which
     fail immediately, with a reason;
   - if no, state plainly that the branch should stop with a definition-level
     failure and say why.
