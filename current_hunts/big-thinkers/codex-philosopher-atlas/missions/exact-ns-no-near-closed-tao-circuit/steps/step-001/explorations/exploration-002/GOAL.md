# Exploration 002 Goal — Packet Language And Canonical Representation Policy

## Objective

Freeze the packet-language policy for this branch.

This exploration must decide, as sharply as the local record permits:

- what the branch should count as a packet object
- what `support` means for Step 1 and Step 2
- whether real-valuedness forces conjugate completion at the object level or
  only at bookkeeping level
- what role helical signs and phase data play in packet identity
- what canonical representation policy later robustness checks must use
- which later changes count as cosmetic canonicalization versus substantive
  modeling choices

The output should be strong enough that Step 1 can write a canonical
packet-language memo and scope-lock sheet without drifting into Step-2
definition writing.

## Success Criteria

- Produce an explicit packet-language recommendation:
  single modes, conjugate-complete pairs, or packet-level / helical-packet
  objects.
- State what is `[VERIFIED]` by the local record versus what is `[INFERRED]`
  or `[PROPOSED]` as a branch policy choice.
- State how conjugate completion, helical bookkeeping, amplitude hierarchy,
  phase conventions, and packet regrouping should be treated under the
  canonicalization policy.
- Separate:
  - true exact symmetries,
  - canonicalization choices,
  - and substantively different modeling moves.
- End with a bounded verdict on whether Step 2 can inherit one fixed packet
  semantics without representation cherry-picking.

## Failure Criteria

- Leaves `support` vague enough that later robustness audits could change the
  object by bookkeeping convenience.
- Treats conjugate completion or helical basis choice as obviously cosmetic
  without justification.
- Uses generic Fourier language without deciding object identity.
- Writes a candidate near-closure definition instead of a representation memo.
