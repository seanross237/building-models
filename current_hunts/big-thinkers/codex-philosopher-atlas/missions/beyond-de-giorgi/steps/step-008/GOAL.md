# Step 8 Goal: Chain Step 1 Freeze One Framework and One Endpoint Before Any Host-Space Ranking

## Mission Context

**Mission:** Beyond De Giorgi — What Structural Property Could Break the NS
Regularity Barrier?

**Active chain:**
`Frozen-Framework Compactness-Rigidity Falsification With Early Endpoint Lock`

**Why this step exists now:** `step-007` killed the prior compactness-rigidity
branch at the host-space package gate and `decision-011` ordered replanning.
Planning run `run-005` selected the current chain as the next canonical path,
but only in a more conservative form. The lesson from the failed branch is
that a fresh host-space audit is not the immediate next move unless the mission
first freezes one coherent ledger: one solution framework, one contradiction
target, one temporal architecture, one bridge to the inherited suitable-weak /
local-energy floor, and one explicit progress currency. This step exists to
make that setup concrete or to kill the branch immediately if no such setup
can be frozen honestly.

## This Step Covers

This strategizer execution covers **Chain Step 1 only**:

- choose one primary solution framework in which the whole branch will be
  judged;
- state one theorem-level endpoint or contradiction line where the branch
  would have to cash out;
- state one temporal architecture for any later minimal object;
- state the exact bridge, if any, to the inherited suitable-weak /
  local-energy floor;
- declare the burden categories that later steps must keep separated;
- declare the progress currency that any later "missing lemma" claim must use;
- and decide whether this frozen setup is coherent enough to justify a later
  host-space audit.

Do **not** rank `L^3`, `\dot H^{1/2}`, or `BMO^{-1}` yet. Do **not** define a
minimal blowup object yet. Do **not** screen rigidity quantities yet. Do
**not** allow a moving setup in which framework, endpoint, or temporal
architecture shifts mid-argument.

## Required Deliverables

Produce all of the following inside `RESULTS.md`:

1. **Framework-and-endpoint memo**
   - list the candidate solution frameworks and contradiction endpoints
     actually considered;
   - choose exactly one primary framework and exactly one theorem-level
     endpoint, or say plainly that no coherent setup exists;
   - label each substantive claim `[VERIFIED]`, `[INFERRED]`, or `[PROPOSED]`.

2. **Temporal-architecture note**
   - state the exact temporal packaging the branch would use for any later
     critical object:
     - forward maximal-lifespan,
     - ancient,
     - normalized near blowup time,
     - or another explicitly named option;
   - explain why that packaging matches the frozen framework and endpoint;
   - reject any setup whose temporal architecture is only symbolic.

3. **Bridge-and-burden ledger**
   - state the exact bridge, if any, from the frozen framework to the inherited
     suitable-weak / local-energy floor;
   - separate the later burden categories explicitly:
     - extraction and stability,
     - symmetry and temporal normalization,
     - pressure or local-energy compatibility,
     - overlap with prior art,
     - and packaging robustness of later rigidity quantities;
   - state the exact progress currency that later contradiction claims must
     use.

4. **Setup verdict**
   - classify the step outcome as one of:
     - `framework frozen, proceed to host-space audit`
     - `setup obstruction, branch should stop before host-space audit`
   - if the setup survives, state exactly what the next step is allowed to vary
     and what is already frozen;
   - if the setup fails, say plainly that the branch should be invalidated at
     the setup stage rather than reopened rhetorically in Step 2.

5. **Source map**
   - name the exact source files used for every framework, endpoint, bridge,
     and burden-ledger claim;
   - distinguish prior mission evidence, run-005 planning evidence, and new
     synthesis.

## Exploration Tasks

Use **2-4 explorations total** unless a kill condition fires early.

### Exploration A: Freeze the candidate setup menu

Tasks:
- identify the smallest serious menu of solution frameworks and theorem-level
  endpoints that are still live after `step-007` and `run-005`;
- reject cosmetic variants that differ only in packaging language;
- state the inherited local-energy baseline that any serious candidate must
  respect.

Success standard:
- the step has a concrete setup menu rather than a drifting Kenig-Merle
  template.

### Exploration B: Stress-test one-ledger coherence

Tasks:
- test whether one framework can carry the extraction, critical-object, and
  later rigidity burdens on a single ledger;
- fix the temporal architecture and the bridge to the inherited local-energy
  floor;
- declare the burden columns and progress currency explicitly.

Success standard:
- either one coherent setup is frozen or the step earns an honest setup
  obstruction.

### Exploration C: Write the setup verdict

Only run this after the surviving setup, if any, is fixed.

Tasks:
- compare the remaining setup options under one standard;
- choose one setup only if it is concrete enough that the next step can audit
  host spaces without reopening framework questions;
- otherwise write the setup obstruction cleanly.

Success standard:
- mission control receives either a frozen setup for the host-space audit or a
  step-level recommendation to kill the branch immediately.

## Kill Conditions

Trigger an early negative conclusion for this step if any of the following
happens:

- no single framework can host the extraction, critical-object, and rigidity
  burdens on one ledger;
- the endpoint, temporal architecture, or bridge changes across candidate
  arguments rather than staying frozen;
- the only way to keep the setup alive is to hide stacked theorem debt inside
  vague language such as "standard Kenig-Merle machinery";
- the progress currency cannot be stated concretely enough to measure a later
  complexity drop;
- the setup survives only by ignoring the inherited suitable-weak /
  local-energy baseline or by postponing its bridge indefinitely.

If a kill condition fires, count that as a successful bounded negative result
for this step and recommend branch invalidation before any host-space audit is
opened.
