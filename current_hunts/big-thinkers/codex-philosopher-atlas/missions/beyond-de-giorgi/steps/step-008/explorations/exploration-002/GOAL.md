# Exploration 002 Goal

## Objective

Stress-test whether the only surviving framework from Exploration 001,
the inherited suitable-weak / Leray-Hopf plus LEI floor, can support one
coherent compactness-rigidity ledger for this chain.

The framework menu is already frozen. Do not reopen it. The question now is
whether this floor can honestly freeze:

- one temporal architecture for any later critical object,
- one exact bridge statement from the frozen framework to later compactness-
  rigidity language,
- the required burden columns,
- and one progress currency for any later contradiction claim.

If not, say plainly that the branch should stop before any host-space audit.

## Required Questions

Answer all of the following from local repository materials:

1. What temporal architecture, if any, is actually supported on the frozen LEI
   floor:
   forward maximal-lifespan,
   ancient,
   normalized near blowup time,
   or another explicitly named option?
2. Does the local record provide an exact bridge from the frozen LEI framework
   to a later critical-element / compactness-rigidity program, or only a
   requirement that such a bridge would be needed?
3. Can the burden categories now be frozen cleanly and separately:
   extraction and stability,
   symmetry and temporal normalization,
   pressure or local-energy compatibility,
   overlap with prior art,
   and packaging robustness of later rigidity quantities?
4. Which progress currency should later contradiction claims be forced to use:
   coercivity, localization burden, compactness debt, regularity demand, or
   another exact option supported locally?
5. Does one coherent setup survive, or does the branch still fail at the setup
   stage because temporal architecture, bridge, or progress currency remain
   symbolic or drifting?

## Working Hypotheses To Test, Not Assume

- The LEI floor is a real baseline, but not yet a compactness-rigidity
  framework with delivered temporal packaging.
- The local record names temporal options only as required future choices, not
  as frozen theorems.
- The bridge to a later critical-element program is still missing rather than
  implicitly available.
- The burden categories are supportable because run-005 freezes them
  explicitly.
- The progress currency can be narrowed to `compactness debt`, because the
  sharpest current failure is still diffuse architecture debt rather than lack
  of an operative coercive mechanism.

Confirm, refine, or reject those hypotheses from source-backed evidence.

## Success Criteria

- Produce a one-ledger coherence table with columns:
  `item`, `actual local support`, `frozen or missing`, `reason`.
- Label major claims `[VERIFIED]`, `[INFERRED]`, or `[PROPOSED]`.
- State one exact temporal-architecture verdict.
- State one exact bridge verdict.
- State one exact progress-currency verdict.
- End with one setup classification only:
  `framework frozen, proceed to host-space audit`
  or
  `setup obstruction, branch should stop before host-space audit`.

## Failure Criteria

- The report silently imports a temporal package that the local record never
  delivers.
- The report treats "would need a bridge" as if it were itself a bridge.
- The report revives critical-space framework talk that Exploration 001 already
  rejected.
- The report claims a progress currency without tying it to the actual failure
  mode on disk.

If failure occurs, say so directly and recommend a setup obstruction.
