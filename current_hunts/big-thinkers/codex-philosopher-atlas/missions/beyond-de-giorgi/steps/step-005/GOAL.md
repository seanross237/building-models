# Step 5 Goal: Chain Step 1 Fix The Architecture, Bottleneck, Protocol, Compatibility, And Novelty Claim

## Mission Context

**Mission:** Beyond De Giorgi — What Structural Property Could Break the NS
Regularity Barrier?

**Active chain:** `Fixed-Protocol Obstruction Audit for Exact NS
Reformulations`

**Why this step exists now:** `step-004` invalidated the prior geometry branch
and sent the mission to replanning. Planning run `run-003` selected the
exact-reformulation audit as the new active chain. Before any Tao-screen memo
or estimate ledger can be trusted, this branch has to lock one inherited
architecture, one exact bad term, one compatible solution class, one
localization protocol, one gain metric, and one bounded candidate family. If
that setup cannot be made concrete, the branch should die early as an
under-specification result rather than drift into algebraic optimism.

## This Step Covers

This strategizer execution covers **Chain Step 1 only**:

- choose exactly one inherited estimate architecture for the audit;
- choose exactly one mission-central bad term inside that architecture;
- name the solution class and theorem-hypothesis compatibility constraints;
- freeze one localization protocol before comparing reformulations;
- define one bookkeeping currency of gain;
- state what is genuinely new relative to established local negatives and
  nearby prior art;
- and produce a bounded reformulation family that is honest enough to justify
  the later Tao screen.

Do **not** treat this step as permission to run Chain Step 2 or Chain Step 3.
Do not test candidate estimates yet. Do not promote a rewrite because it looks
structurally pretty before the architecture and bookkeeping rules are fixed.

## Required Deliverables

Produce all of the following inside `RESULTS.md`:

1. **Architecture-and-bottleneck screening memo**
   - choose exactly one inherited architecture;
   - choose exactly one named bad term inside that architecture;
   - explain why this is the right audit target for the current chain rather
     than a rediscovery of an already-closed negative;
   - state what later success would mean in this exact setup.

2. **Solution-class and compatibility note**
   - name the solution class in which the localized estimate is supposed to
     operate;
   - record any theorem-hypothesis compatibility constraints needed for the
     chosen architecture to make sense;
   - identify one explicit circularity risk or package mismatch that would
     kill the branch if ignored.

3. **Frozen localization protocol**
   - fix the cutoff placement, localization region, and where projection,
     commutator, or shell costs will be paid;
   - explain why this protocol is the fairest audit setup for the chosen bad
     term;
   - include a short note on what would count as an illegitimate mid-branch
     protocol change later.

4. **Gain-currency and candidate-family table**
   - choose one bookkeeping currency of gain:
     derivative count, integrability or exponent improvement, coefficient
     shrinkage, or a strictly weaker hypothesis in the same closure scheme;
   - define a bounded family of at most `2-3` exact reformulations of the same
     nonlinearity;
   - optionally name one paired hybrid route only if it still targets the same
     bad term in the same architecture;
   - for each item, state its intended leverage point and first obvious reason
     it may be cosmetic.

5. **Novelty and Step-2 readiness memo**
   - state what is genuinely new relative to:
     - the established De Giorgi sharpness record,
     - the pressure-route negatives,
     - and nearby prior art relevant to the chosen architecture;
   - say whether the candidate family is now concrete enough for the next
     step's Tao discriminator screen;
   - if not, recommend early branch invalidation rather than continuation.

## Kill Conditions

Trigger an early negative conclusion for this step if any of the following
happens:

- no single inherited architecture and bottleneck term can be fixed without
  collapsing into already-settled negatives or into a comparison protocol too
  vague to audit;
- the chosen solution class and localization package require assumptions that
  already smuggle in the hoped-for gain or make the branch effectively
  equivalent to the original regularity problem;
- the candidate family cannot be bounded to `2-3` honest reformulations plus
  at most one same-target hybrid route;
- the supposed novelty claim disappears once local project history and nearby
  prior art are stated explicitly.

If a kill condition fires, count that as a successful obstruction result for
this step and recommend branch invalidation or replanning rather than
continuing.
