# Refined Chain 01 - Falsification-First Exact-NS Property Audit

## Central Premise

This branch should not try to produce a polished cross-architecture atlas unless at least one candidate survives a hard theorem-facing stress test. Its honest purpose is narrower:

- identify whether any exact Navier-Stokes property is genuinely lost under Tao-style averaging,
- survives the actual solution and localization regime needed by one proof architecture,
- and yields a typed gain at one named bottleneck that the averaged route does not match.

If that cannot be shown, the correct output is a bounded obstruction or hybrid-handoff memo, not a forced "primary branch."

## Verifiable Result Target

Either:

- one live candidate packet inside a shared comparison class, containing a Tao discriminator, prior-art overlap account, bottleneck insertion point, gain currency, and first missing lemma; or
- one bounded obstruction memo showing that all audited candidates fail because of Tao-insensitivity, localization collapse, prior-art absorption, architecture mismatch, or non-operative bottleneck contact; or
- one hybrid-handoff memo showing that a candidate only becomes live after changing architecture, propagated object, or comparison level.

## Why This Refined Chain Is Better

The original chain had the right instinct but too much discretionary ranking. This refined version keeps the comparison-first discipline while making positive promotion harder to earn and negative conclusions more precise.

## Scope Lock

Before the chain is considered active, it must freeze all of the following:

- one comparison level:
  identity-level,
  transport-level,
  estimate-level,
  localized bottleneck-interaction level,
  or contradiction-mechanism level;
- one exact NS observable or structural statement to test;
- one intended solution class and localization regime;
- one theorem-facing bottleneck family;
- one gain currency:
  derivative count,
  exponent gap,
  coercive remainder,
  localization stability,
  compactness input,
  or propagation time;
- one worked pass case and one worked fail case for the Tao discriminator;
- one reopening rule for previously killed branches;
- one overlap ledger distinguishing:
  global negatives,
  architecture-specific negatives,
  and mere style warnings.

If these cannot be fixed concretely, stop and write an under-specified-filter memo.

## Ordered Steps

### Step 1 - Calibrate the Tao discriminator at one stable level

- Depends on: none.
- Action: define exactly what "destroyed by averaging" means at the chosen comparison level.
- Action: specify the exact observable, the averaged-model comparison statement, and the evidence standard for passing the Tao screen.
- Action: produce one nontrivial pass example and one nontrivial fail example that remain distinct under small reformulation.
- Expected output: a calibration memo that actually constrains later decisions.
- Kill condition: if the pass/fail pair is not robust, terminate with the under-specified-filter memo.

### Step 2 - Build typed candidate cards from bounded sources

- Depends on: Step 1.
- Action: shortlist at most four candidate families from mission-live sources.
- Action: for each card, record:
  claimed exact-NS feature,
  comparison class,
  intended solution framework,
  localization demands,
  target bottleneck family,
  and nearest prior-art mechanism.
- Action: if a previously killed branch is reopened, require one explicit substantive change in comparison level, architecture, or bottleneck exposure.
- Expected output: a typed candidate ledger rather than a slogan list.
- Kill condition: if every candidate is either recycled without substantive change or only identity-deep, end with a recycled-portfolio obstruction.

### Step 3 - Run overlap and compatibility gates before ranking

- Depends on: Step 2.
- Action: audit each candidate for mechanism-level overlap with prior art and with earlier mission dead ends.
- Action: test whether the candidate survives the intended solution class and localization regime.
- Action: classify each candidate immediately as:
  directly compatible,
  hybrid-only,
  prior-art absorbed,
  or dead on contact.
- Expected output: a compatibility-and-overlap screen that removes false survivors early.
- Kill condition: if no candidate remains directly compatible after this gate, close with a compatibility or overlap obstruction memo.

### Step 4 - Attach each survivor to one exact bottleneck and one gain currency

- Depends on: Step 3.
- Action: for each surviving candidate, name one primary architecture and one exact bad term, closure gap, compactness obstruction, or contradiction input where it must act.
- Action: define the minimum measurable gain that would count as non-cosmetic progress in the chosen currency.
- Action: forbid broad architecture labels from standing alone without a typed insertion point.
- Expected output: a bottleneck-attachment sheet with an actual measure of progress.
- Kill condition: if a candidate cannot name one precise insertion point and one minimum gain, demote it immediately.

### Step 5 - Demand a hard artifact before promotion

- Depends on: Step 4.
- Action: for every still-live candidate, require at least one of:
  prototype insertion into a representative estimate,
  toy-model or localized stress test,
  or a failed lemma whose failure sharply identifies the obstruction.
- Action: reject candidates that remain persuasive only at the descriptive level.
- Expected output: a hard evidence packet for each candidate that survives narrative screening.
- Kill condition: if all candidates remain documentary after this step, close with a documentary-success obstruction.

### Step 6 - Compare only where comparison is earned

- Depends on: Step 5.
- Action: rank candidates only if they share the same comparison class, primary architecture, and gain currency.
- Action: otherwise report bucketed outcomes:
  live lead,
  hybrid handoff,
  bounded obstruction,
  or unresolved non-promotion.
- Action: allow zero promoted branches without treating that as process failure.
- Expected output: either a narrow same-class ranking or a bucketed result.
- Kill condition: if finishing the chain requires apples-to-oranges ranking, replace ranking with buckets.

### Step 7 - Close with the narrowest earned claim

- Depends on: Step 6.
- Action: state exactly which comparison level, candidate class, solution regime, bottleneck family, and gain currency were audited.
- Action: state exactly what Tao discrimination did and did not buy.
- Action: record the first missing lemma only for candidates that survived the hard-artifact gate.
- Action: record non-reentry rules only for branches closed by a hard obstruction rather than by loose preference.
- Expected output: one of three endings:
  live candidate packet,
  bounded obstruction memo,
  or hybrid-handoff memo.
- Kill condition: if the conclusion outruns the evidence packet, narrow it until it is fully earned.

## Probability Assessment

**Probability that this refined chain yields a presentable result: 0.76**

The most likely presentable output is a rigorous obstruction memo. A positive survivor is possible, but only if it survives a much sharper evidentiary ladder than the original chain imposed.
