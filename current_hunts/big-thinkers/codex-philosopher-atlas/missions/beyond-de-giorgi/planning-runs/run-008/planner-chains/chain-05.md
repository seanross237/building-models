# Chain 05 - Critical Element With Pre-Admitted NS Firewall

## Central Premise

Concentration-compactness is worth revisiting only as a second-stage architecture. This branch is live only if one exact NS-only firewall quantity has already survived an earlier screen and can be defined on the compactness class before rigidity optimism begins.

## Verifiable Result Target

Either:

- one theorem-like critical-element program naming one host space, one compactness package, one firewall quantity, and one exact missing lemma; or
- one obstruction memo locating the first fatal break at extraction,
  stability,
  definability of the firewall quantity on the compactness class,
  or failure of contradiction even under almost periodicity.

## Why This Chain Is Meaningfully Different

This chain is global and contradiction-based. It assumes blowup only after a concrete exact-NS lever already exists, unlike the earlier compactness branches that hoped a rigidity input would appear later.

## Ordered Steps

### Step 1 - Audit one extraction package before any rigidity talk

- Depends on: none.
- Action: choose one host space and require it to clear an explicit ledger:
  local theory,
  perturbation or stability,
  extraction or profile decomposition,
  symmetry normalization,
  pressure or local-energy compatibility,
  and a usable compactness notion.
- Expected output: an extraction-package memo with one admitted space and one precise missing theorem, if any.
- Kill condition: if no host space clears that full ledger, stop with a compactness-floor obstruction.

### Step 2 - Formulate the minimal object before promoting the firewall

- Depends on: Step 1.
- Action: define the hypothetical critical element, its normalization, and the exact almost-periodicity or compactness information available.
- Action: forbid any firewall quantity that is not well-defined on this class.
- Expected output: a critical-object dossier in theorem-like form.
- Kill condition: if the minimal object cannot be stated with a clear dependency graph to known results, stop and record a formulation obstruction.

### Step 3 - Admit only one firewall quantity that closes on the compactness class

- Depends on: Step 2.
- Action: choose at most one firewall quantity imported from Chains 01-04.
- Action: require four things:
  definability on the critical class,
  Tao relevance as a secondary discriminator,
  one contradiction mechanism,
  and one complexity drop relative to arbitrary finite-energy solutions.
- Expected output: a firewall-admission memo with one promoted quantity or a no-firewall verdict.
- Kill condition: if no quantity closes on the actual compactness class, terminate with a rigidity-input obstruction.

### Step 4 - Run the contradiction test under almost periodicity

- Depends on: Step 3.
- Action: write one explicit contradiction pathway showing how the firewall quantity rules out the critical element.
- Action: separate what is already known from the first genuinely narrower missing lemma.
- Expected output: a contradiction blueprint or a typed failure memo.
- Kill condition: if the missing lemma is as hard as full regularity or if almost periodicity gives no extra leverage, downgrade immediately to obstruction.

### Step 5 - Close in theorem-like conditional form

- Depends on: Step 4.
- Action: state the result as:
  assumptions,
  exact known inputs,
  exact missing lemma,
  and what follows if that lemma is proved.
- Action: if negative, name the first honest failure point precisely.
- Expected output: either a compactness-plus-firewall program or a sharply located obstruction memo.
- Kill condition: if the final output is only a polished roadmap rather than a theorem-like conditional or obstruction statement, narrow it until the debt is explicit.
