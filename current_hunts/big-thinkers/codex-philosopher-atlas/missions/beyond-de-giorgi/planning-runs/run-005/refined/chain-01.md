# Refined Chain 01 - Frozen-Framework Compactness-Rigidity Falsification With Early No-Host Gate

## Central Premise

Treat the compactness-rigidity branch primarily as an audited falsification
program, not as a free-standing positive blueprint.

A live forward route exists only if one host-space / solution-framework pair
supports all of the following in one coherent package:

- a real extraction statement rather than a hoped-for one,
- a theorem-like minimal blowup object with explicit temporal architecture,
- one rigidity quantity defined on that actual class,
- and a contradiction step whose first missing input is genuinely smaller than
  the full regularity problem.

If no such package exists, the correct result is an obstruction memo stating
where the branch fails and whether the failure is no-host, no-frozen-object,
no-rigidity, or no-complexity-drop.

## Verifiable Result Target

Either:

- an audited no-host verdict under one frozen framework;
- a theorem-like conditional program naming one host space, one solution
  framework, one critical-element class, one admissible rigidity quantity, and
  one genuinely narrower missing lemma; or
- a stacked obstruction memo showing that the compactness-rigidity route lacks a
  coherent audited starting point even before the final contradiction.

## Why This Refined Chain Is Worth Running

Run-005 evidence already suggests that the branch is more likely to close
negatively than positively. That is not a reason to drop it. It is a reason to
run it honestly as the portfolio's best tool for certifying exactly where the
Kenig-Merle-style dream still breaks for 3D Navier-Stokes.

## Scope Lock

Before host spaces are compared, the chain must fix all of the following:

- one primary solution framework;
- the temporal architecture that any minimal object would have to live in:
  forward maximal-lifespan, ancient, normalized near blowup time, or another
  explicitly named option;
- burden categories that are not merged onto one fake scale:
  extraction/stability,
  symmetry and temporal normalization,
  pressure or local-energy compatibility,
  and prior-art overlap;
- the rule that a stacked multi-theorem debt counts as a decisive failure, not
  as "close except for one theorem";
- the policy that Tao-sensitivity is mandatory but secondary:
  it cannot rescue a mathematically non-operative route, and it cannot be
  ignored once a route becomes operational;
- the progress currency for Step 4:
  coercivity, localization burden, compactness debt, regularity demand, or
  another explicitly named currency.

If these items cannot be fixed concretely, stop and write a setup obstruction.

## Ordered Steps

### Step 1 - Freeze one solution framework before ranking host spaces

- Depends on: none.
- Action: choose one primary framework in which the whole branch will be judged.
- Action: state what bridge, if any, is required to connect that framework to
  the inherited suitable-weak / LEI floor.
- Action: reject any setup that compares host spaces across incompatible
  frameworks while pretending they are commensurable.
- Expected output: a framework memo naming one solution class and one temporal
  architecture.
- Kill condition: if no single framework can host the later extraction,
  minimal-object, and rigidity burdens on one ledger, stop and record a
  framework obstruction.

### Step 2 - Re-audit host spaces with separated burden columns

- Depends on: Step 1.
- Action: compare `L^3`, `\dot H^{1/2}`, and only if still justified
  `BMO^{-1}` inside the frozen framework.
- Action: for each space, record separately:
  local theory,
  large-data stability or perturbation,
  extraction or profile input,
  symmetry and temporal normalization,
  pressure or local-energy compatibility,
  and prior-art overlap.
- Action: treat a diffuse multi-theorem architecture gap as a decisive failure,
  not as a near miss.
- Action: use the run-005 host-space audit as evidence to be confirmed or
  narrowed, not as context to be ignored.
- Expected output: a package-audit table plus either one surviving host-space /
  framework pair or an explicit no-host verdict.
- Kill condition: if every candidate fails by stacked debts rather than one
  narrow missing ingredient, stop and write the no-host obstruction.

### Step 3 - Define the minimal blowup object only if a single pair survives

- Depends on: Step 2.
- Action: state the minimizing quantity, solution class, lifespan type, time
  normalization, symmetry group, and pressure conventions.
- Action: name the exact compactness or almost-periodicity statement the chosen
  extraction package delivers, rather than an aspirational version of it.
- Action: reject any formulation that imports new unproved structure beyond what
  Step 2 already admitted.
- Expected output: a theorem-like critical-object dossier with explicit temporal
  architecture and no hidden bridges.
- Kill condition: if the critical object is only formal because the compactness
  statement is missing or mismatched to the chosen class, stop and record a
  formulation obstruction.

### Step 4 - Screen rigidity quantities with a dual gate

- Depends on: Step 3.
- Action: list only rigidity quantities that are defined on the actual
  critical-element class.
- Action: first apply the mathematical-operability gate:
  definability,
  stability under the compactness limit,
  non-cosmetic role in the contradiction,
  and one explicit mechanism such as coercivity, monotonicity, unique
  continuation, or exclusion of a named scenario.
- Action: only after that, apply the mission gate:
  record what exact NS-specific feature the quantity uses and whether Tao-style
  averaging appears to destroy that feature.
- Action: reject candidates that merely restate familiar templates without a new
  operational lever.
- Expected output: one promoted rigidity quantity and at most one backup, each
  with an operability note and a Tao-relevance note.
- Kill condition: if no quantity survives the operability gate, or if the only
  survivors are Tao-compatible in the relevant sense, stop and write a rigidity
  obstruction.

### Step 5 - Demand a genuine complexity drop in a declared currency

- Depends on: Step 4.
- Action: run the promoted quantity through the critical element line by line.
- Action: state where compactness, temporal architecture, pressure structure,
  and regularity requirements enter.
- Action: identify the first missing input and measure the claimed reduction in
  the declared progress currency.
- Action: if the missing input really requires simultaneous gains in
  compactness, compatibility, and coercivity, classify that honestly as a
  stacked architectural obstruction rather than a fake single lemma.
- Expected output: either a conditional contradiction schema with a real
  complexity drop or a typed obstruction memo.
- Kill condition: if the alleged missing lemma is effectively the whole problem,
  or if the route only works by hiding multiple debts inside one label, stop and
  downgrade to obstruction.

### Step 6 - Close with the narrowest earned claim

- Depends on: Step 5.
- Action: write the final result in theorem-like form.
- Action: if positive, name exactly one host space, one framework, one
  critical-element class, one rigidity quantity, one missing lemma, and one
  conditional consequence.
- Action: if negative, state whether the first honest break is:
  no coherent host space,
  no operational critical object,
  no admissible rigidity quantity,
  or no genuine complexity drop.
- Action: allow the final obstruction to be stacked if that is what the audited
  record supports; do not force it into a cosmetically sharp single-break story.
- Expected output: a final-comparison-ready chain that is either a narrow
  conditional program or a precise obstruction map.
- Kill condition: if the final claim cannot be written without overstating the
  coherence of the package, narrow it further until it is fully earned.

## Success Standard

Do not count the branch as successful because it resembles a Kenig-Merle
program. It succeeds only if it either:

- isolates one audited host-space / framework pair and one real rigidity route
  with a genuinely narrower missing lemma; or
- proves, in a disciplined and localized way, that no such coherent package is
  presently available.

## Probability Assessment

**Probability that this refined chain yields a presentable result: 0.76**

The most credible endpoint is a no-host or no-rigidity obstruction memo. A live
positive program is possible only after the refined chain survives gates that
the original version did not enforce.
