# Chain 05 - Critical Element Program With Pre-Admitted Firewall Quantity

## Central Premise

Compactness-rigidity is not a primary search branch. It becomes live only if an upstream chain first isolates one exact-NS firewall quantity that is theorem-facing, definable on the compactness class, and stable enough to survive extraction and localization. Without that firewall, the correct output is an immediate no-entry verdict.

## Why This Chain Is Meaningfully Different

This is the only contradiction-architecture chain. It does not search for a new mechanism inside the PDE. It asks whether a mechanism already admitted elsewhere can survive concentration-compactness, almost-periodic normalization, and contradiction without collapsing back into the full theorem.

## Ordered Steps

### Step 1 - Enforce the firewall admission gate before any host-space work

- Depends on: one upstream chain having produced a concrete firewall candidate.
- Action: write the exact firewall quantity, the endpoint it supports, and why it is specific to the true NS system rather than to generic harmonic analysis.
- Action: require that the quantity be definable on the intended compactness class and stable under limits.
- Expected output: either a firewall-admission memo or an immediate no-entry verdict.
- Kill condition: if no concrete firewall packet exists, stop immediately with a firewall-missing obstruction.

### Step 2 - Freeze one host space, one framework, and one normalization package

- Depends on: Step 1.
- Action: choose one host space only, such as `L^3` or `\dot H^{1/2}`, and one solution framework.
- Action: audit local theory, stability, pressure compatibility, extraction, and compatibility of the firewall quantity with that package.
- Action: freeze one normalization package for the hypothetical minimal object.
- Expected output: a host-space and normalization audit.
- Kill condition: if the host space lacks a coherent extraction package or the firewall is not definable there, stop with a host-space obstruction.

### Step 3 - Define the critical object in theorem-like form

- Depends on: Step 2.
- Action: state the minimizing quantity, symmetry normalization, compactness statement, and exact class of the hypothetical minimal blowup object.
- Action: ensure the firewall survives in that class with no hidden new assumptions.
- Expected output: a critical-object dossier.
- Kill condition: if the critical element cannot be formulated without importing diffuse extra assumptions, stop with a formulation obstruction.

### Step 4 - Transport the firewall through localization and compactness

- Depends on: Step 3.
- Action: check that the firewall quantity survives weak limits, profile extraction, and the localized contradiction package actually in use.
- Action: expose where nonlocal terms, pressure restoration, or normalization shifts might destroy the firewall.
- Expected output: a compactness-survival memo.
- Kill condition: if the firewall is not stable under the extraction package, stop with a survival obstruction.

### Step 5 - Run the contradiction path and charge complexity honestly

- Depends on: Step 4.
- Action: insert the firewall into the almost-periodic or compactness-normalized setting.
- Action: identify exactly where contradiction should arise and what the first missing lemma is.
- Action: reject any route whose first missing lemma is equivalent to full regularity or to rebuilding the entire upstream mechanism.
- Expected output: a contradiction-path memo plus a first-missing-lemma statement.
- Kill condition: if there is no genuine complexity drop, stop with a no-complexity-drop obstruction.

### Step 6 - Close with the narrowest earned blueprint or obstruction

- Depends on: Step 5.
- Action: if positive, write one theorem-like conditional blueprint naming the host space, framework, critical-element class, firewall quantity, contradiction line, and first narrower missing lemma.
- Action: if negative, classify failure as missing firewall, host-space incoherence, critical-object formulation failure, compactness-survival failure, or no real complexity drop.
- Expected output: a comparison-ready compactness blueprint or obstruction memo.
- Kill condition: if Tao sensitivity is the main reason the branch still looks attractive, downgrade it to obstruction.
