# Separate True Symmetries Canonicalization And Substantive Object Changes Before Claiming Support Stability

When `support` is mission-central, freeze the smallest object that keeps exact
bookkeeping honest before making any stability claim. In packet branches, that
often means promoting packet identity above single-mode bookkeeping and treating
conjugate completion as mandatory canonical representation rather than as the
primary object itself.

Then split representation moves into three classes:

- true symmetries of the chosen object
- canonicalization choices fixed once up front
- substantive object changes that create a different model

Until that split is frozen, packet regrouping, helical-basis changes, and even
conjugate-completion language should not be assumed cosmetic. This is a common
failure mode behind false claims that an object is "stable up to
representation."
