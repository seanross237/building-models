# The Concrete Node: A Codeable Node-First Foundation

## Design Philosophy

The two existing proposals correctly identify the node as the stable unit and agents as ephemeral contributors. This proposal takes the next step: it specifies exactly what gets built. Every structure described here should be directly translatable into Python dataclasses, directory layouts, and validation functions.

The operating principle: if you cannot write a test for it, it is not a contract.

---

## What A Node Is

A node is a directory on disk that represents one bounded unit of work. It has exactly five subdirectories, a finite set of well-known files, and a single status value drawn from a closed enum. A node does not know what kind of agent will work on it. It does not care whether it is a planning node, a worker node, or something that does not exist yet. It only knows: here is my goal, here is my status, here is what has been produced so far.

A node is the smallest thing you can point at and ask: "what happened here?"

## What A Node Owns

A node owns exactly these things and nothing else:

1. **Its goal** -- immutable input written at creation time
2. **Its status** -- a single value from a closed set, written only by the orchestrator
3. **Its artifacts** -- files under `output/` produced by agent runs
4. **Its orchestrator control state** -- files under `for-orchestrator/` read by the orchestrator to make decisions
5. **Its run history** -- logs and metadata under `system/` that record what happened
6. **Its children** -- subdirectories under `children/` that are themselves nodes

A node does not own its agent. It does not own its prompt. It does not own its model. Those are supplied externally at spawn time.

## What An Agent Run Is

An agent run is a single process invocation that reads from a node, does work, writes artifacts back into the node, and exits. The relationship is:

- **Node : Agent Run = Hotel Room : Guest.** The room has a contract (bed, bathroom, checkout time). The guest uses the room and leaves. A different guest can use it next. The room's records persist; the guest's memory does not.

An agent run has no persistent identity. If the orchestrator needs more work done on a node, it spawns a fresh run. The new run reconstructs context entirely from the node's files. There is no session continuity, no hidden state, no memory beyond what is written to disk.

Multiple agent runs may contribute to a single node over its lifetime (librarian, then planner, then mid-plan-evaluator, etc.), but at most one run is active at any time.

## The Node Directory Layout

Every node, regardless of what kind of work it does, has this exact structure:

```
<node>/
  input/
    goal.md                     # immutable, written at creation
    parent-instructions.md      # alternative to goal.md for child nodes
    context.md                  # optional: accumulated context from prior steps
    knowledge.md                # written by retrieval pass (replaces the long filename)
    instructions-<role>.md      # generated by orchestrator before each spawn
  output/
    plan.md                     # if this node planned
    review.md                   # if the plan was reviewed
    final-output.md             # the terminal artifact
    state.md                    # running state, updated by agents
    escalation.md               # if the node escalated
  for-orchestrator/
    status                      # the single source of truth for node status
    mode                        # what kind of agent should run next
    eval-decision               # decision from evaluator: continue|replan|escalate
    cost.json                   # accumulated cost for this node and descendants
    waiting                     # marker: background computation in progress
    computation-result          # marker: background computation finished
  system/
    runs/                       # one subdirectory per agent run
      run-001/
        meta.json               # pid, role, model, started_at, finished_at
        loop.jsonl              # per-turn log of the agent loop
        usage.json              # token and cost summary for this run
        stdout.log
        stderr.log
    artifacts/                  # generated scripts, intermediate files
    jobs/                       # background job metadata
  children/
    step-01-<slug>/             # child nodes, same structure recursively
    step-02-<slug>/
```

Notable changes from v1:

- `this-nodes-current-status` becomes `status`. Shorter. Same function.
- `agent-mode` becomes `mode`. Same.
- `retrieved_relevant_knowledge_from_library.md` becomes `knowledge.md`. The old name was descriptive but unwieldy.
- Agent run logs move from flat files in `system/` to structured subdirectories under `system/runs/`. This matters because a single node may have many runs over its lifetime, and they should not clobber each other.
- `cost.json` is a new first-class file. Cost tracking is not optional.

## The Status Enum

Five statuses. No more.

```python
class NodeStatus(str, Enum):
    PENDING   = "pending"
    RUNNING   = "running"
    DONE      = "done"
    FAILED    = "failed"
    TIMED_OUT = "timed_out"
```

That is the entire node-level state model. Everything else the current v1 tracks as a status -- `retrieving`, `needs_review`, `reviewing`, `needs_decision`, `deciding`, `approved`, `executing`, `evaluating`, `waiting_comp`, `escalated`, `cancelled` -- is either:

- **A sub-phase of RUNNING**, derivable from inspecting what files exist and what child states are active
- **A specific outcome** that can be represented as DONE + an outcome marker, or FAILED + a reason

### How Sub-Phases Work

The orchestrator does not need a status to know what to do next. It needs to inspect the node's current artifacts and children:

| Condition | What the orchestrator does |
|---|---|
| status=PENDING, no run active | Spawn retrieval, then spawn the role in `mode` |
| status=RUNNING, `knowledge.md` missing, run active | Wait for retrieval to finish |
| status=RUNNING, `plan.md` exists, no review yet | Route to review if needed, or auto-approve |
| status=RUNNING, `plan.md` approved, next step unspawned | Create next child |
| status=RUNNING, active child exists | Wait |
| status=RUNNING, all children terminal | Spawn evaluator or synthesizer |
| status=RUNNING, `final-output.md` exists | Transition to DONE |
| status=RUNNING, `escalation.md` exists | Transition to DONE with outcome=escalated |

This is the same logic the current v1 orchestrator already implements. The difference is that instead of encoding each of these conditions as a named status, the orchestrator derives them from file inspection. This keeps the status model stable even as we add new sub-phases.

### Completion Outcomes

DONE is a status. What the node produced is an outcome:

```python
class NodeOutcome(str, Enum):
    COMPLETED  = "completed"    # final-output.md exists and satisfies contract
    ESCALATED  = "escalated"    # escalation.md exists
    CANCELLED  = "cancelled"    # parent cancelled this node
```

The outcome is written to `for-orchestrator/outcome` when the node transitions to DONE. This replaces having `complete`, `escalated`, and `cancelled` as separate statuses.

### Why This Matters

The v1 system has 15 statuses. Every new agent type or workflow variation risks adding more. With 5 statuses, the state machine is closed. New agent types add new file artifacts and new orchestrator inspection logic, but they never add new statuses. The status enum is a protocol constant, not an extension point.

## The Node Data Model

```python
@dataclass
class NodeSpec:
    """Written once at creation. Immutable."""
    node_id: str              # e.g. "step-01-analyze-structure"
    parent_id: str | None     # None for root
    created_at: str           # ISO 8601
    goal: str                 # the task description
    mode: str                 # initial agent mode: "planner", "worker", etc.
    depth: int                # 0 for root
    complexity: int           # 1-10, from parent's plan
    importance: int           # 1-10, from parent's plan

@dataclass
class NodeState:
    """Mutable. Derived from files on disk, not stored as a single blob."""
    status: NodeStatus
    outcome: NodeOutcome | None
    run_count: int            # how many agent runs have happened
    active_run_pid: int | None
    cost_usd: float           # accumulated cost including descendants
    children: list[str]       # child node IDs in order
```

`NodeSpec` maps to the immutable files written at creation (`goal.md`, `mode`, the node directory name). `NodeState` is never stored as a single file -- it is reconstructed by reading the node directory. This is important: the node's state is the directory, not a database row.

## The Run Record

Every agent run produces a structured record:

```python
@dataclass
class RunRecord:
    run_id: str               # "run-001", "run-002", etc.
    role: str                 # "librarian", "planner", "worker", etc.
    model: str                # "anthropic/claude-sonnet-4"
    started_at: str
    finished_at: str | None
    pid: int
    exit_reason: str          # "artifacts_satisfied", "max_iterations", "error", "zombie_killed"
    iterations: int
    prompt_tokens: int
    completion_tokens: int
    cost_usd: float
    artifacts_produced: list[str]  # relative paths of files written
```

This is stored as `system/runs/run-NNN/meta.json`. The run number increments monotonically. When the orchestrator needs to understand a node's history, it reads these records in order.

### Why Per-Run Directories Matter

In v1, each spawn overwrites `active-run.json` and appends to shared logs. If a node gets three runs (librarian, planner, mid-plan-evaluator), the history is tangled. With per-run directories, each run is isolated, comparable, and independently inspectable. This is what makes A/B testing possible later: you can compare run-003-variant-A against run-003-variant-B on the same node structure.

## The Three Handoff Validations

The debate concluded that structural validation should happen at handoff points, not through a schema registry. There are exactly three handoff points:

### 1. Node Creation Handoff

When the orchestrator creates a child node, validate:

```python
def validate_node_creation(node_dir: Path) -> None:
    """Called after creating a child. Fails loudly if broken."""
    assert (node_dir / "input" / "goal.md").exists() or \
           (node_dir / "input" / "parent-instructions.md").exists(), \
           "Node has no goal"
    assert (node_dir / "for-orchestrator" / "status").exists(), \
           "Node has no status file"
    assert (node_dir / "for-orchestrator" / "mode").exists(), \
           "Node has no mode file"
    status = (node_dir / "for-orchestrator" / "status").read_text().strip()
    assert status == "pending", f"New node has status '{status}', expected 'pending'"
```

### 2. Run Completion Handoff

When an agent run finishes (process exits or is killed), validate that the run produced what was needed:

```python
REQUIRED_ARTIFACTS = {
    "librarian":           ["input/knowledge.md"],
    "planner":             ["output/plan.md"],
    "worker":              ["output/final-output.md"],
    "plan-reviewer":       ["output/review.md"],
    "mid-plan-evaluator":  ["for-orchestrator/eval-decision"],
    "synthesizer":         ["output/final-output.md", "for-orchestrator/eval-decision"],
}

ESCALATION_EXEMPT = {"planner", "worker", "plan-decider", "mid-plan-evaluator", "synthesizer"}

def validate_run_completion(node_dir: Path, role: str) -> bool:
    """Returns True if the run satisfied its contract. Does not raise."""
    if role in ESCALATION_EXEMPT and (node_dir / "output" / "escalation.md").exists():
        return True
    required = REQUIRED_ARTIFACTS.get(role, [])
    return all((node_dir / path).exists() for path in required)
```

### 3. Node Completion Handoff

When the orchestrator is about to transition a node to DONE, validate the terminal state:

```python
def validate_node_completion(node_dir: Path, outcome: str) -> None:
    """Called before writing DONE. Fails loudly if the outcome is unsupported."""
    if outcome == "completed":
        assert (node_dir / "output" / "final-output.md").exists(), \
               "Cannot complete without final-output.md"
    elif outcome == "escalated":
        assert (node_dir / "output" / "escalation.md").exists(), \
               "Cannot escalate without escalation.md"
    elif outcome == "cancelled":
        pass  # no artifact requirement
    else:
        raise ValueError(f"Unknown outcome: {outcome}")
```

That is three validation functions. Not sixty. They check exactly what the downstream consumer needs to read.

## Cost Caps

Cost is tracked per-node in `for-orchestrator/cost.json`:

```json
{
  "self_usd": 0.42,
  "children_usd": 1.87,
  "total_usd": 2.29,
  "cap_usd": 10.00
}
```

The orchestrator updates this file after every run completes by reading the run's `usage.json` and summing children. The cap is inherited from the mission config and can be overridden per-node.

Cost enforcement is hard:

```python
def check_cost_cap(node_dir: Path) -> bool:
    """Returns False if the node has exceeded its cost cap."""
    cost_file = node_dir / "for-orchestrator" / "cost.json"
    if not cost_file.exists():
        return True
    cost = json.loads(cost_file.read_text())
    return cost["total_usd"] <= cost["cap_usd"]
```

If `check_cost_cap` returns False, the orchestrator transitions the node to FAILED with a reason of "cost_cap_exceeded". It does not ask the model whether to continue. This is a hard kill.

## Sandboxed Execution

The debate concluded that sandboxing the environment is better than maintaining a tool registry that tries to enumerate allowed operations. In practice, this means:

- **Write boundary**: agents can only write inside their own node directory. This is enforced in the tool layer (already implemented in v1's `_resolve_write_path`).
- **Read boundary**: agents can read anything inside the repo. This is permissive by design -- agents need context.
- **Execution boundary**: shell commands run with the node directory as cwd. Timeout limits are enforced by the runtime, not by the model.
- **No network access**: agents cannot make HTTP calls except through the runtime's own API client. This is enforced by not providing network tools.

The tool set remains open to extension. New tools are added to the registry and enabled per-role in config. The sandbox constraints apply uniformly regardless of which tools are enabled.

## The Orchestrator As Sole State Authority

The orchestrator is the only process that writes to `for-orchestrator/status`. Agents never touch it. This is already true in v1 and must remain true.

The orchestrator's responsibilities:

1. **Bootstrap**: create the root node from `mission-goal.md`
2. **Poll**: walk all nodes, read their status and artifacts
3. **Decide**: based on the 5-status model + file inspection, determine the next action
4. **Act**: spawn agent runs, create child nodes, transition statuses
5. **Enforce**: cost caps, timeouts, zombie detection, validation
6. **Log**: every status transition, every spawn, every validation failure

The orchestrator communicates with agents exclusively through files. There is no RPC, no message queue, no shared memory. The filesystem is the message bus.

### The Immutable Log

Every orchestrator action is appended to `orchestrator.jsonl`. This log is append-only and never modified. Each entry:

```json
{
  "ts": "2026-04-01T12:34:56Z",
  "event": "status_change",
  "node": "tree/root/children/step-01-analyze",
  "from_status": "running",
  "to_status": "done",
  "outcome": "completed",
  "reason": "final-output.md exists and validated",
  "run_id": "run-003",
  "cost_usd_at_transition": 1.42
}
```

## What Must Be Hard vs. Soft

### Hard (enforced in code, tested offline)

| Rule | Enforcement point |
|---|---|
| Status only transitions through legal moves | `write_status()` with expected-value check |
| DONE requires validated artifacts | `validate_node_completion()` |
| Cost cap kills the node | `check_cost_cap()` before every spawn |
| Only one active run per node | PID check before spawn |
| Agents cannot write outside their node | `_resolve_write_path()` |
| Zombie processes get killed | `check_zombie()` with strike counter |
| Every transition is logged | `log_event()` inside `write_status()` |
| Run records are never overwritten | Monotonic run numbering under `system/runs/` |
| Eval decisions are consumed and deleted | Orchestrator unlinks after reading |
| Node creation is validated | `validate_node_creation()` |

### Soft (model judgment, quality-sensitive but not protocol-critical)

| Behavior | Why it is soft |
|---|---|
| How a plan is decomposed | Different decompositions can all be valid |
| How a review is worded | Review quality matters but is not checkable by code |
| How a synthesis combines outputs | Judgment call |
| How an evaluator decides continue/replan/escalate | Judgment call, but the value is validated |
| How a worker reasons through a task | The contract is the output, not the process |

## Extensibility: Adding New Agent Types

Adding a new agent type (say, a `verifier` that checks mathematical claims) requires:

1. **A system prompt** at `stuff-for-agents/verifier/system-prompt.md`
2. **A role entry** in `REQUIRED_ARTIFACTS` specifying what it must produce (e.g., `["output/verification.md"]`)
3. **A config entry** specifying its model and enabled tools
4. **Orchestrator logic** for when to spawn it (e.g., "after worker completes on math nodes, spawn verifier before transitioning to DONE")

It does not require:

- A new status value
- A new directory structure
- A new validation framework
- Changes to the node data model
- Changes to the run record format

The node stays the same. The status enum stays the same. The validation functions gain one more entry in the `REQUIRED_ARTIFACTS` dict. The orchestrator gains a few lines of inspection logic. That is the extensibility contract.

## A/B Testability

The per-run directory structure is the foundation for A/B testing. Here is how a future experiment works:

1. **Define the variant** in mission config: "for nodes with mode=worker and depth <= 2, use model X instead of model Y"
2. **The orchestrator passes the variant** to the agent runner at spawn time
3. **The run record captures the variant**: `meta.json` includes a `variant` field
4. **Comparison happens at the node level**: same node structure, same goal, same validation -- different run records

The node is the comparison boundary. Two runs on comparable nodes can be compared on:

- cost (from `usage.json`)
- time (from `meta.json` timestamps)
- success rate (from node outcomes)
- artifact quality (from `output/final-output.md`, assessed by a separate judge)
- iteration count (from `meta.json`)

Because run records are isolated in their own directories, you can run the same node with variant A, reset it to PENDING, and run it again with variant B without any file collision.

## Biggest Risks

### 1. Orchestrator complexity explosion

The 5-status model pushes decision logic into file inspection rather than status transitions. If the inspection logic is not well-structured, the orchestrator becomes a pile of conditionals that is hard to reason about. **Mitigation**: organize the orchestrator as a decision table, not a chain of if/else. Each row in the table maps (status, file conditions) to an action. Test each row independently.

### 2. File inspection races

The orchestrator polls. An agent writes files. If the orchestrator reads a half-written file, or reads `plan.md` before `status` has been updated, it could make a wrong decision. **Mitigation**: the orchestrator always reads status first and only inspects files relevant to the current status. Agents write artifacts before signaling completion (they do not write status at all -- the orchestrator detects completion from artifacts). The polling interval provides a natural debounce.

### 3. Cost tracking drift

If cost accounting has bugs -- a run's cost is not recorded, or children's costs are not aggregated -- the cap becomes unreliable. **Mitigation**: cost is updated synchronously after each run exits, before the orchestrator takes any other action on that node. The cost file is validated (total = self + children) on every read.

### 4. Run directory proliferation

A node that gets retried many times could accumulate dozens of run directories. **Mitigation**: this is actually fine. Disk is cheap, and the history is valuable. But set a hard retry limit (e.g., 5 runs per node) enforced in code. After 5 runs without satisfying the artifact contract, the node transitions to FAILED.

### 5. The "just one more file" creep

Over time, there is pressure to add more well-known files to the node contract. Each addition is small, but collectively they make the node harder to understand. **Mitigation**: any new protocol file must be added to the validation functions and the canonical contracts doc. If it is not worth validating, it is not worth making canonical.

## What This Proposal Does Not Cover

This proposal stays node-focused. The following are explicitly deferred:

- **Workflow-level abstractions**: if nodes need reusable process recipes (e.g., "always do retrieval then planning then review"), that is a layer above nodes. This proposal does not prevent it, but does not design it.
- **Cross-node communication**: nodes communicate through the tree (parent reads child outputs). There is no peer-to-peer messaging in this design. If that becomes necessary, it should be a separate proposal.
- **Parallel child execution**: the current design executes children sequentially. Parallel execution is possible but changes the orchestrator's inspection logic. Deferred.
- **Agent prompt engineering**: how prompts are written and assembled is important but orthogonal to the node foundation.

## Implementation Priority

If someone sits down to code this, the order should be:

1. Define `NodeStatus` and `NodeOutcome` enums
2. Implement the three validation functions
3. Restructure the node directory layout (especially `system/runs/`)
4. Implement `cost.json` tracking and cap enforcement
5. Simplify `for-orchestrator/` filenames (status, mode, outcome)
6. Refactor the orchestrator's `handle_*` methods into a decision table
7. Write offline tests for every row of the decision table
8. Migrate existing missions to the new layout (or accept a clean break)
