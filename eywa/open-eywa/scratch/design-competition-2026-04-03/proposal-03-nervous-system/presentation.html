Node contract

Inputs are task, parent state, instruction bundle, tree vars, node vars, and context slice.

Trace packet

Write node id, parent id, variables, tools, outputs, score, cost, and elapsed time on every run.

Invariant

Take any node from the middle of the tree. It should still be the same machine at its core.

Learning

How does Bonsai choose variables?

Bonsai sits around Eywa. It learns from run history, benchmarks, and task shape, then returns a variable profile.

What it reads

Task tags, embeddings, scores, cost, time, and the variables used in previous runs.

What it learns

First heuristics. Then a variable policy model once the benchmark set is clean enough.

What it writes

Recommended variables, benchmark promotions, and task-family performance summaries.

Change control

How does Potter improve code safely?

Potter may simplify and modularize the implementation, but only after dry runs prove that validated behavior stays identical.

Allowed improvements

Lower complexity, smaller modules, stronger boundaries, and cleaner traces.

Allowed differences

Execution time may move a little. Semantics may not.

Potter output

A pass or fail plus a report on modularity, intuitiveness, and robustness.

Open questions

What still needs to be decided?

The proposal is concrete enough to build, but these contracts need a final decision before optimization can scale.

Which variables are tree level and which are node level?

Freeze the first schema so future runs can compare like with like.

What is the minimum benchmark taxonomy?

Choose the task tags that best predict the right variable profile.

What must the dry run machine fully simulate?

Potter cannot change code until replay is deterministic enough to trust.

When does a run graduate into the benchmark bank?

Set a promotion rule for verifiable runs and keep the history separate.

Freeze the trace packet and variable schema first. Then Eywa, Bonsai, and Potter can move independently without losing the contract.
