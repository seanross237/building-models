# Section 1 — Doc Update H — `02-subgraph-types.md` (Option 3)

## What I changed

- Rewrote the intro and the "What a type contains" section so a subgraph type is described as a **fully-fixed bundle** (shape + per-slot templated meta prompts + variable defaults), with the seed `delegate_fanout_synthesize` example replaced by the contract's exact JSON (envelope tags, `buddy_bound_variables`, per-slot `meta_prompt`, etc.).
- Added a short "Template variables in meta prompts" subsection introducing auto-bound vs buddy-bound variables and pointing the reader at `subgraph-type-contract.md` for the full list and resolution rule.
- Inserted a new "Option 3 commitment" section between "What a type does NOT contain" and "The seeded library" — covers the commitment, the four reasons (reproducibility, clean attribution, clean A/B, clearer Scientist role), the cost, and the `{buddy_free_instruction}` escape hatch.
- Updated both seed examples (`just_execute` and `delegate_fanout_synthesize`) to match the contract's seed examples verbatim (envelope, `buddy_bound_variables: []`, full templated meta prompts).
- Updated "What a type does NOT contain", "How the library grows", "Type versioning", "Content-addressed type identity", "DAG-only — no loops" (left as-is, double-checked), and "Variants as experiments" so they all describe the bundle model and the Scientist-publishes-bundles workflow without any Option 1 phrasing.

## Every judgment call

- **Where I put the new "Option 3 commitment" section.** Placed it directly after "What a type does NOT contain" and before "The seeded library (day one)". Rationale: the negative-space section already ends on "no Buddy-authored per-slot instructions", so the next natural beat is "and here's why we made that trade." Putting it before the seed examples means the reader hits the commitment before they see the templated meta prompts and immediately understands why those prompts live in the bundle.
- **Length of the Option 3 commitment section.** Aimed for ~14 lines including the four reason bullets and the cost paragraph. The brief said 6–12 lines but I went a hair over because each of the four reasons needed to be its own short bullet — collapsing them into prose would have made the section harder to scan and less load-bearing.
- **`{buddy_free_instruction}` escape hatch wording.** Mentioned it as a single parenthetical sentence at the end of the cost paragraph. The brief said "brief mention, not detailed" — I wrote "future variant ... could restore open-ended per-turn instruction authoring for novel cases — not in the seed library, added by the Scientist if and when it's worth the loss of attribution cleanliness." That makes the trade-off explicit (you give up clean attribution to get back open-ended authoring) without going into binding-type vocabulary or contract specifics.
- **`contract_name` / `contract_version` description.** Called them "envelope tags identifying this file as a `subgraph_type` document; used by the loader and validator." Kept it short to match the existing field-explanation style — these are mechanical, not load-bearing concepts for a reader trying to understand the design.
- **`role` field description.** Pulled the contract's "Used for display, debugging, and grouping in learning surfaces. Not the prompt — the prompt lives in `meta_prompt`." line almost verbatim because the brief explicitly said to clarify it's a display/debug label, and that contract sentence is exactly that clarification.
- **`meta_prompt` field description.** Wrote "templated system prompt for this slot. Non-empty string. May contain template variables that the runtime substitutes at materialization time. This is what gets rendered into the worker's actual prompt." I deliberately said "system prompt" rather than "prompt" because in the eywa runtime that's the standing distinction; it should be clearer to future readers that this is the role/instruction layer, not user content.
- **`buddy_bound_variables` description.** Mentioned both the explicit and the auto-computed mode, plus the empty-list interpretation. Kept it to one short paragraph in the field list rather than a full sub-subsection — the contract has the full validation rules.
- **Template variables subsection placement.** Put it as a `###` subsection inside "What a type contains" rather than as its own top-level `##` section. Rationale: it's an explanation of the `meta_prompt` field, not a parallel concept to the slot model. Treating it as a sibling of "What a type contains" would have implied it's its own concept the reader has to internalize separately.
- **Auto-bound vs buddy-bound brevity.** Listed three example auto-bound patterns (`{task.text}`, `{inputs.<slot_label>.output}`, `{prior_turns[N].terminal_output}`) and gave one Scientist-style example for buddy-bound (`{plan}`/`{review}`). Did not enumerate `{inputs.<slot_label>.summary}` or `{prior_turns[N].summary}` here — those are in the contract and the design doc shouldn't duplicate the full vocabulary. The brief said "point the reader at `subgraph-type-contract.md` for the full list and resolution rule — don't duplicate the contract's precision," and I followed that.
- **"Where the library lives" section.** Left it untouched — the directory path is unchanged and the contract confirms that location. Double-checked.
- **"DAG-only — no loops" section.** Left mostly untouched. Made one small change: "picks it again with revised instructions" → "picks it again with revised bindings", because under Option 3 Buddy doesn't author instructions on a turn, he authors `context_bindings`. That's the only Option 1 phrasing I found in this section.
- **"How the library grows" section.** Reformatted from a numbered 1–6 list to a bulleted 6-item list and rewrote each bullet to emphasize that the Scientist crystallizes the **whole bundle** (not just a "type" of "shape"). Added the explicit point that the bench compares bundles directly because there is no Buddy authoring inside the bundle to disentangle. The brief said 4–8 bullets; I have 6.
- **"Type versioning" section.** Rewrote to lead with "changing the meta_prompt on any slot — or any other part of the bundle — produces a new content hash." Added "rewords the synthesizer's meta prompt" as an alternative example next to the existing "adds a light review slot" example, to make concrete that prompt rewording is a versioning event.
- **"Content-addressed type identity" section.** Rewrote the second sentence of the original section to spell out exactly what the hash now captures (shape + every slot's meta_prompt + buddy_bound_variables + per-slot variable defaults). Added "Editing a single character in any meta prompt produces a new hash" because that line is the load-bearing one for the learning-surfaces motivation.
- **"Variants as experiments" section.** Restructured the prose into a 4-bullet list of axes (meta prompt wording, variable defaults, slot shape, Buddy policy). The original was a single paragraph that mixed "buddy policies, variable defaults, edge structures." Bullets are more legible and the new "meta prompt wording" axis is the headline change so it deserves its own bullet.
- **Keeping "Why only two" almost untouched.** None of the bullets in that section used Option 1 phrasing — they all said "Plan, then execute" / "Research in parallel" etc., which describes Buddy's sequencing, not slot authoring. The argument is unchanged. I left it as-is.
- **"What a type does NOT contain" — adding "no actual model outputs" back.** The original had "Actual model outputs" as a bullet; I kept it because it's still true under Option 3 and removing it would be a regression.

## Things I left alone that might warrant reconsideration

- The "Plan subagent's critique" reference in "Why only two" stays. It's a historical pointer to an earlier design conversation; under Option 3 the argument is still valid (shipping too many types creates folklore), so rewriting it would be churn.
- The `delegate_fanout_synthesize_v2` example in "Type versioning" still describes "adds a light review slot" — that's a shape change, not a meta prompt change. I added "or rewords the synthesizer's meta prompt" as an alternative, but I kept the shape-change example because the original had it and it's still a valid axis. A future cleaner pass could lead with the meta-prompt-rewording case since that's the most novel under Option 3.
- "How the library grows" still uses the word "graveyard" for the loser folder. I kept it because it was in the original and it's vivid; I have no opinion on whether that folder name will be the literal one when it gets implemented.
- The `delegate_fanout_synthesize` seed example in this doc is identical to the contract's seed example, including the slightly weak "handle the Nth piece" partition guidance that Builder G flagged in his notes. I deliberately did not "fix" it here — the contract is the source of truth, and if the seed type itself needs to grow a partition variable, that's a Section 2 change that should land in both places at once, not a quiet drift here.
