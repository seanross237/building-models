# Strategy 001: Freeze One Canonical Helical Packet Object, Then Force or Kill the Bottleneck

## Objective

Turn the mission question

```text
can low-leakage Tao-like packet dynamics in exact NS be pushed toward global
one-sign coherence while the desired activation edges still require positive
heterochiral dependence?
```

into one theorem-facing packet/sign test object and drive it to one of the mission's terminal outcomes:

1. positive helical bottleneck,
2. counterexample: low-leakage low-sign-defect packet realization exists,
3. canonicity failure,
4. model-level failure: singleton sign clue does not survive packet lift.

This strategy is not allowed to drift into generic packet menus, turbulence folklore, or a return to the already-killed singleton exact circuit.

## Methodology

### Phase 0: Canonical Object Gate

Start from the best available packet-level exact object and decide whether there is one canonically defensible helical packet/sign object for this mission.

The preferred object must freeze:

- packet families,
- conjugate/sign bookkeeping,
- desired Tao-like role graph,
- one leakage functional,
- one global sign-coherence defect,
- one desired-edge heterochirality defect.

Required deliverables:

- one preferred packet/sign object and at most one backup,
- a notation table,
- one explicit statement of what is canonical and what is still choice-dependent,
- one clear positive theorem target and one clear counterexample target.

Phase 0 kill conditions:

- If the object changes under harmless packet refinements, representative choices, or bookkeeping conventions, stop with `canonicity failure`.
- If the only faithful object collapses back to singleton exact-mode structure, stop with `model-level failure`.

### Phase 1: Packet Lift of the Sign Signal

Using the preferred packet/sign object, test whether the singleton clue survives:

- low leakage should push toward low global minority-helicity participation,
- but the desired Tao-like activation edges may still require positive heterochiral dependence.

Required deliverables:

- one computed or tightly justified relation among `Leak`, the global sign-defect quantity, and the desired-edge heterochirality quantity,
- one exact statement of whether the singleton signal survives, weakens, or disappears.

Phase 1 kill conditions:

- If the sign-sensitive squeeze disappears under the packet lift, stop with `model-level failure`.
- If the relation depends on arbitrary packet bookkeeping choices, stop with `canonicity failure`.

### Phase 2: Quantitative Bottleneck Test

Attack one precise obstruction statement or one explicit counterexample target.

Each exploration in this phase must begin with one concrete target:

- one inequality candidate,
- one incompatibility statement,
- one minimizing-sequence / extremizer story,
- or one explicit low-leakage counterexample candidate.

Acceptable outputs are:

- a lower bound on desired-edge heterochirality under low leakage,
- an implication from low leakage to low global sign-defect,
- an incompatibility of simultaneously small `Leak`, `SD_part`, and `SD_target`,
- or a faithful packet realization where all three are small enough to kill the route.

### Phase 3: Mandatory Adversarial Screen

Any apparent bottleneck or counterexample must be stress-tested against:

- near-homochiral packet families,
- mixed-sign target edges with suppressed spectator leakage,
- symmetry choices that might trivialize the sign functionals,
- and any hidden dependence on the first failed clustered exact-mode packet model.

This phase must answer explicitly whether the surviving claim is:

- exact-NS-specific,
- genuinely helical/sign-sensitive,
- and not merely a scalarized restatement of generic leakage minimization.

### Phase 4: Final Verdict

Finish with exactly one named mission outcome:

1. positive helical bottleneck,
2. counterexample: low-leakage low-sign-defect packet realization exists,
3. canonicity failure,
4. model-level failure: singleton sign clue does not survive packet lift.

The final report must make the implication unambiguous:

- pursue,
- kill,
- or reformulate.

## Cross-Phase Rules

1. By the end of Phase 0 there must be one preferred packet/sign object and at most one backup.
2. One theorem target or one counterexample target per exploration.
3. Exact-NS helical-triad fidelity is non-negotiable. Any reduced packet object must state preserved and discarded exact structure explicitly.
4. Quantitative content is mandatory. Every accepted claim must point to a named measurable quantity.
5. Do not reopen De Giorgi, epsilon-regularity, pressure, host-space, or hidden-precursor routes under new language.
6. Use the singleton sign-defect prototype only as a clue; it is not theorem evidence by itself.
7. Every substantial claim in the final report must be tagged `[VERIFIED]`, `[COMPUTED]`, `[CHECKED]`, or `[CONJECTURED]`.
8. Maintain a Direction Status Tracker in `REASONING.md` using `OPEN / PROMISING / CLOSED / EXHAUSTED`.
9. Early stopping is correct if canonicity fails cleanly, the singleton signal dies under packet lift, or a quantitative obstruction/counterexample survives adversarial review.

## Validation Criteria

This strategy succeeds if it delivers all of the following:

- one frozen packet/sign object,
- one explicit leakage functional,
- one explicit global sign-defect quantity,
- one explicit desired-edge heterochirality quantity,
- either a quantitative obstruction statement or an explicit counterexample,
- a mandatory adversarial screen,
- an unambiguous terminal verdict in one of the mission's named categories.

This strategy is exhausted if:

- the object cannot be canonized sharply enough,
- the sign-sensitive singleton clue does not survive packet lift,
- the quantitative bottleneck/counterexample question is answered and survives adversarial review,
- or the only remaining uncertainty is clearly a sharper next-strategy problem rather than more of the same search.
