# Step 13 Goal: Chain Step 1 Calibrate The Endpoint Window And Burden Ledger

## Mission Context

**Mission:** Beyond De Giorgi — What Structural Property Could Break the NS
Regularity Barrier?

**Active chain:** `Quantified Reformulation Audit With Anti-Repackaging, Early
Endpoint Exchange, And Interface-Aware Budgeting`

**Why this step exists now:** Planning run `run-009` replaced the invalidated
stochastic-transport branch with the reformulation-audit chain as the canonical
next branch. That choice was not permission to start testing reformulations
immediately. The branch is live only if it first fixes the theorem-facing
ledger in which all later gains and debts will be charged.

This step is therefore the chain's calibration gate. It must freeze the
endpoint window, the baseline burden, the allowed ledger currencies, the
comparability rule, the anti-repackaging classifier, the solution-class and
localization regime, and the prior-art audit format before any candidate family
is allowed to claim progress. If that scope lock cannot be made concrete, the
branch should stop here with an under-specified-ledger obstruction rather than
continue into candidate derivations.

## This Step Covers

This strategizer execution covers **Chain Step 1 only**:

- choose one primary endpoint family and one backup endpoint family on the
  inherited LEI / CKN floor;
- freeze one theorem-facing baseline burden and one admissible notion of gain
  in that burden language;
- define one typed burden ledger with explicit allowed currencies:
  derivative count,
  locality,
  cancellation,
  integrability,
  pressure recoverability,
  admissibility,
  compactness input,
  and scaling compatibility;
- define one same-currency comparability rule and one transfer-lemma rule for
  any cross-currency claim;
- define the anti-repackaging classification rule with the three required
  outputs:
  `equivalent redistribution`,
  `ambiguous redistribution`,
  and `potentially live mechanism change`;
- freeze one intended solution class and one localization regime for later
  candidate testing;
- freeze one nearest-prior-art audit format naming the closest known
  reformulation and the exact claimed mathematical delta; and
- decide whether the branch has earned continuation into Chain Step 2.

Do **not** build the candidate slate yet. Do **not** derive exact reformulated
equations yet. Do **not** run bottleneck insertion tests yet. Do **not** let a
wide menu of endpoints, burdens, or reformulation families survive this step.

## Required Deliverables

Produce all of the following inside `RESULTS.md`:

1. **Scope-lock sheet**
   - freeze one primary endpoint family and one backup endpoint family;
   - freeze one theorem-facing baseline burden;
   - freeze one admissible gain notion in that burden language;
   - freeze one typed burden ledger with the allowed currencies listed above;
   - freeze one same-currency comparability rule and one transfer-lemma rule;
   - freeze one solution class and one localization regime;
   - freeze one nearest-prior-art audit format that later candidate packets
     must satisfy.

2. **Endpoint memo**
   - explain why the primary endpoint is the right first theorem-facing window
     for this branch;
   - explain why the backup endpoint is the correct reserve instead of a wider
     menu;
   - state what kinds of reformulation gains would count at this interface and
     what kinds would be merely decorative;
   - label substantive claims `[VERIFIED]`, `[INFERRED]`, or `[PROPOSED]`.

3. **Burden-ledger memo**
   - define the baseline burden in concrete theorem-facing language;
   - define each allowed burden currency and how later steps should record
     gains and debts in it;
   - state when same-currency comparison is sufficient;
   - state when a transfer lemma is mandatory rather than optional.

4. **Anti-repackaging and prior-art memo**
   - define the exact test separating
     `equivalent redistribution`,
     `ambiguous redistribution`,
     and `potentially live mechanism change`;
   - define the minimum evidence needed for a candidate to avoid immediate
     prior-art absorption;
   - define the audit format later steps must use to name the closest known
     reformulation and the exact claimed delta.

5. **Step verdict**
   - classify the outcome as one of:
     - `scope lock passed, proceed to candidate slate and localized equation audit`
     - `under-specified-ledger obstruction, branch should stop before candidate audit`
   - if the step passes, state exactly what is now frozen for Chain Step 2;
   - if the step fails, say plainly that the branch should not continue into
     candidate reformulation work.

6. **Source map**
   - name the exact source files used for every endpoint, burden, solution-
     class, localization, anti-repackaging, prior-art, and verdict claim;
   - distinguish planning-run evidence, prior mission evidence, and new
     synthesis created in this step.

## Exploration Tasks

Use **2-4 explorations total** unless a kill condition fires early.

### Exploration A: Freeze the endpoint window

Tasks:
- choose one primary endpoint family on the inherited theorem floor;
- choose one backup endpoint family only;
- justify why this pair is neither artificially rigid nor too wide to govern
  later candidate comparison;
- state what theorem-facing insertion question these endpoints are meant to
  discipline.

Success standard:
- the step exits with one primary endpoint and one backup endpoint, not a
  menu.

### Exploration B: Freeze the burden ledger

Tasks:
- define one baseline burden the chain is trying to improve;
- define one admissible gain notion in that burden language;
- define the allowed ledger currencies and the same-currency rule;
- define when a transfer lemma is required for cross-currency claims.

Success standard:
- later steps can compare gains and debts mechanically instead of narratively.

### Exploration C: Freeze the anti-repackaging and prior-art audit

Tasks:
- define the three-way anti-repackaging classifier;
- define what later steps must show to avoid prior-art absorption;
- define the format for naming the closest known reformulation and the exact
  mathematical delta;
- state what kind of novelty will be rejected as identity-level or equivalent
  redistribution.

Success standard:
- later candidate packets can be killed quickly if they only rename the old
  burden.

### Exploration D: Freeze the solution-class and localization regime, then write the verdict

Only run this after the endpoint and ledger are concrete.

Tasks:
- choose one intended solution class and one localization regime for later
  candidate testing;
- state whether those choices are compatible with the endpoint window and
  burden ledger;
- decide whether the branch is now concrete enough to move to Chain Step 2;
- write the final step verdict and the exact frozen inputs for the next step.

Success standard:
- mission control receives either a precise Step-2 entry protocol or a clean
  under-specified-ledger obstruction.

## Kill Conditions

Trigger an early negative conclusion for this step if any of the following
happens:

- the step cannot reduce the endpoint choice to one primary and one backup
  family;
- the baseline burden remains vague, multi-valued, or detached from the
  inherited LEI / CKN bottlenecks;
- the ledger currencies are not explicit enough to support same-currency
  comparison;
- the step permits cross-currency improvement claims without naming a transfer
  lemma requirement;
- the anti-repackaging test cannot distinguish identity-level redistribution
  from a potentially live mechanism change;
- the prior-art audit format is too weak to expose absorbed reformulations; or
- the intended solution class or localization regime remains too under-specified
  to govern later localized equation work.

If a kill condition fires, count that as a successful bounded negative result
for this step and recommend branch invalidation before Chain Step 2.
