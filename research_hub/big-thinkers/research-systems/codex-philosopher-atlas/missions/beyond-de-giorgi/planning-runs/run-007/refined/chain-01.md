# Refined Chain 01 - Operational Tao-Discriminator Audit with Early Compatibility Gates

## Central Premise

Treat this branch as a truth-constrained triage and obstruction program, not as an atlas that is allowed to rank loosely related exact-NS themes by narrative appeal.

The branch should ask a narrower question:

- is there a specific exact Navier-Stokes observable or structural statement,
- at one fixed comparison level,
- that is genuinely lost or washed out under Tao-style averaging,
- survives the actual solution class and localization regime,
- and buys a measurable improvement at one named theorem bottleneck?

If not, the correct output is a bounded obstruction or hybrid-handoff memo, not a forced shortlist.

## Verifiable Result Target

Either:

- one or more live candidates inside a shared comparison class, each with a precise Tao discriminator, a named bottleneck, a quantitative gain metric, and a first missing lemma; or
- a bounded obstruction memo showing that the audited candidate classes fail because of Tao-insensitivity, compatibility failure, localization collapse, prior-art absorption, or non-operative endpoint cash-out; or
- a hybrid-handoff memo showing that a candidate only becomes live after changing architecture, solution framework, or propagated object.

## Why This Refined Chain Is Better

The original chain had the right instinct but too soft a court of evidence. This refined version keeps the Tao filter, the bounded search space, and the willingness to end negatively, while forcing common comparators before any ranking language is allowed.

## Scope Lock

Before the chain is considered active, it must freeze all of the following:

- one comparison level:
  identity-level,
  transport-level,
  coercive-estimate-level,
  localized bottleneck-interaction level,
  or blowup-scenario level;
- one exact NS observable or statement to be tested against averaging;
- one intended solution class and localization regime;
- one theorem bottleneck family that counts as the cash-out target;
- one bookkeeping currency of gain:
  derivative count,
  exponent or integrability improvement,
  coercive remainder shrinkage,
  localization stability,
  or propagation time;
- one passing and one failing worked example under the admissibility rule;
- three separate exclusion ledgers:
  earned global negatives,
  architecture-specific negatives,
  and style warnings.

If these cannot be fixed concretely, stop and write an under-specified-filter memo.

## Ordered Steps

### Step 1 - Calibrate the Tao discriminator

- Depends on: none.
- Action: state exactly what "destroyed by averaging" means at the chosen comparison level.
- Action: name the exact observable, exact averaged-model comparison statement, and exact evidentiary standard for saying the candidate is Tao-sensitive.
- Action: give one worked example that clearly passes and one that clearly fails.
- Expected output: a calibration memo that can actually separate admissible from inadmissible candidates.
- Kill condition: if the Tao test cannot cleanly distinguish at least one pass case from one fail case, close with the under-specified-filter memo.

### Step 2 - Build typed candidate cards, not one flat atlas table

- Depends on: Step 1.
- Action: shortlist at most four candidates.
- Action: assign each candidate a comparison class recording:
  solution framework,
  localization demands,
  endpoint family,
  and nearest prior-art family.
- Action: require each card to name:
  the claimed NS-only feature,
  the comparison level where averaging destroys it,
  the proof-relevance hypothesis,
  and the likely overlap risk.
- Action: reject candidates that are only slogan-level, only identity-level, or already absorbed by known negatives with no new cash-out path.
- Expected output: a typed candidate ledger with evidence cards rather than one-line verdicts.
- Kill condition: if no candidate survives this carding stage, end with a bounded negative memo.

### Step 3 - Run compatibility and localization gates before any ranking

- Depends on: Step 2.
- Action: for each surviving candidate, test immediately whether it lives in the intended solution class and survives the localization regime needed by the bottleneck family.
- Action: record whether the candidate is:
  directly compatible,
  compatible only after architecture change,
  or dead on contact.
- Action: if a candidate is only representation-level at this stage, require one narrower structural lemma that would promote it beyond rhetoric.
- Expected output: an early feasibility memo that prevents dead candidates from gaining narrative momentum.
- Kill condition: if every candidate is dead on contact or architecture-changing only, close with a compatibility obstruction or hybrid-handoff memo.

### Step 4 - Attach each survivor to one exact bottleneck template

- Depends on: Step 3.
- Action: assign each survivor one primary architecture and one exact bad term or bottleneck interaction.
- Action: allow a secondary interpretation only if it still serves the same bottleneck template.
- Action: define the minimum measurable gain that would count as progress in the declared bookkeeping currency.
- Expected output: an insertion map from candidate to bottleneck, gain metric, and first missing lemma.
- Kill condition: if no candidate reaches a named bottleneck with a non-cosmetic gain target, end with an endpoint obstruction memo.

### Step 5 - Force a hard artifact for every live candidate

- Depends on: Step 4.
- Action: require one of the following for each still-live candidate:
  a prototype insertion into the bottleneck estimate,
  a representative failed model lemma,
  or a full bottleneck cost ledger showing exactly where the hoped-for gain is spent.
- Action: reject candidates that survive only as well-described stories.
- Expected output: a hard evidence packet for each live or failed route.
- Kill condition: if every candidate remains purely documentary after this step, stop and write a documentary-success obstruction memo.

### Step 6 - Record failure severity and overlap honestly

- Depends on: Step 5.
- Action: classify each candidate by primary and optional secondary failure mode:
  Tao-insensitive,
  compatibility-blocked,
  localization-collapse,
  prior-art absorbed,
  endpoint mismatch,
  or circular missing lemma.
- Action: also assign severity:
  hard death,
  hybrid handoff,
  or unresolved but non-promoted.
- Action: update the overlap note so a rerun of a known dead family is not mistaken for a new lead.
- Expected output: a severity-aware obstruction ledger.
- Kill condition: if all candidates land in hard death or prior-art absorption, close negatively without forced ranking.

### Step 7 - Compare only where comparison is earned

- Depends on: Step 6.
- Action: rank candidates only if they share the same primary architecture and the same gain currency.
- Action: otherwise report bucketed outcomes:
  live lead,
  hybrid handoff,
  bounded obstruction,
  or unresolved non-promotion.
- Action: promote zero, one, or more candidates according to evidence; do not force a top-two output.
- Expected output: either a narrow ranked list inside one real comparison class or a bucketed portfolio result.
- Kill condition: if the branch needs apples-to-oranges comparison to finish cleanly, replace ranking with buckets.

### Step 8 - Close with the narrowest earned claim

- Depends on: Step 7.
- Action: state exactly what comparison level, candidate class, solution framework, bottleneck family, and gain metric were audited.
- Action: state exactly what Tao-discrimination did and did not buy.
- Action: if the result is negative, say whether the obstruction came from the Tao test, compatibility gates, bottleneck insertion, or comparability rules.
- Expected output: one of three acceptable endings:
  a live candidate packet,
  a bounded obstruction memo,
  or a hybrid-handoff memo.
- Kill condition: if the closing claim reaches beyond the audited class or evidence packet, narrow it until it is fully earned.

## Probability Assessment

**Probability that this refined chain yields a presentable result: 0.71**

The most likely presentable output is a rigorous obstruction or hybrid-handoff memo. A positive survivor is possible, but only if it clears a much sharper standard than the original chain imposed.
