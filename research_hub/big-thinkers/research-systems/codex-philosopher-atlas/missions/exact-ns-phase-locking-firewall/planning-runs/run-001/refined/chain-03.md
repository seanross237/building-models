# Chain 03 - Scoped Intrinsic Phase/Coherence Audit

## Central premise

The branch should test a precise question, not a slogan: within a formally
declared class of exact phase/coherence functionals, does any object survive a
stated intrinsicity test without desired-channel labels, packet-sheet
dependence, or frozen transfer annotations? Honest success can be either a
promoted intrinsic object or a scoped negative over that declared class. It is
not a license to claim that no intrinsic phase object exists in any form.

## Verifiable result target

Either:

- one exact phase/coherence functional that is intrinsic under the declared
  equivalence relation, together with its exact identity or exact unresolved
  expansion status; or
- a decision-grade scoped negative memo of the form:
  "within candidate class `C` under equivalence test `E`, no intrinsic
  survivor exists."

## Prior context to carry forward

- Earlier work showed packet-level objects can still be informative without yet
  being intrinsic enough for theorem-facing use.
- The mission forbids a phase object that becomes meaningful only after
  selecting a desired subgraph or channel by hand.
- This branch is an intrinsicity guardrail. It must not quietly collapse into
  another packet-canonicity audit.

## Ordered steps

### Step 1 - Declare the candidate class and the equivalence test

- Depends on: none.
- Actions:
  - define a closed candidate class `C` to be audited, for example a finite
    grammar or basis of exact-support functionals built from amplitudes,
    relative phases, and interaction weights, with no desired-channel, packet,
    or fixture labels in the raw definition;
  - state the formal equivalence relation `E` explicitly, covering the allowed
    exact symmetries, conjugate completion, helical sign relabeling, gauge
    conventions, and normalization changes;
  - define three statuses up front:
    intrinsic,
    canonically representable but non-intrinsic,
    and inadmissible.
- Expected output:
  - one scope note defining `C`,
  - one explicit equivalence test `E`,
  - one admissibility checklist.
- Kill condition:
  - if `C` or `E` cannot be stated precisely enough for a third party to apply,
    stop with a process memo saying the branch is not yet audit-ready.

### Step 2 - Populate a neutral candidate basis inside `C`

- Depends on: Step 1.
- Actions:
  - generate the candidate family from the declared grammar or basis rather
    than from at most three planner-favored examples;
  - allow examples only as illustrations of subfamilies already covered by `C`;
  - record possible theorem relevance only as an optional note, never as an
    admissibility filter;
  - exclude any candidate whose formula directly names desired channels,
    packet sheets, or inherited fixture labels.
- Expected output:
  - one candidate catalog or reduced basis with formulas and class membership.
- Kill condition:
  - if the neutral generator yields only inadmissible candidates, issue a
    scoped negative at the level of class `C`; do not escalate to a universal
    mission negative.

### Step 3 - Run the formal intrinsicity audit

- Depends on: Step 2.
- Actions:
  - test each candidate under `E` and record whether its value is invariant,
    merely representation-dependent, or inadmissible;
  - require either a proof of invariance or an explicit countertransformation
    showing dependence on forbidden bookkeeping;
  - treat packet-sheet dependence, desired-channel dependence, or frozen
    transfer annotations as immediate failure of intrinsicity.
- Expected output:
  - one audit matrix with status labels and short justifications for each
    candidate.
- Kill condition:
  - if no candidate is invariant under `E`, close the branch with the scoped
    negative:
    no intrinsic survivor exists within `C` under `E`.

### Step 4 - Classify dynamic usefulness without redefining existence

- Depends on: Step 3.
- Actions:
  - for each intrinsic survivor, derive its exact balance law if it closes
    cleanly;
  - if the exact ledger is open or ugly, record that honestly as
    "intrinsic but dynamically unresolved" rather than killing the object;
  - classify each survivor as:
    closed exact law,
    exact but open expansion,
    or static intrinsic functional.
- Expected output:
  - one dynamics ledger or expansion note per intrinsic survivor,
  - one downstream-question note for any survivor that appears theorem-useful.
- Kill condition:
  - do not kill a survivor solely because its dynamics are not elegant. Kill
    only if the object ceases to be exact or well-defined.

### Step 5 - Run diagnostic audits on fixtures and off-template probes

- Depends on: Step 4.
- Actions:
  - evaluate intrinsic survivors on inherited fixtures such as `F_SL`, `F_SS`,
    and `F_DT`, plus at least one off-template or perturbative probe chosen to
    test whether the object only tracks old packet defect scores;
  - use this step to measure interpretability, redundancy, and sensitivity, not
    to re-decide intrinsicity;
  - explicitly note when a survivor is highly correlated with earlier packet
    gates versus when it captures new structure.
- Expected output:
  - one diagnostic table across inherited and off-template probes,
  - one note on whether the survivor seems genuinely phase/coherence-sensitive
    or mostly a re-expression of prior packet metrics.
- Kill condition:
  - diagnostics do not kill a survivor unless they reveal hidden dependence on
    forbidden annotations that should have failed Step 3.

### Step 6 - Close with a scoped promotion or scoped negative

- Depends on: Step 5.
- Actions:
  - if at least one intrinsic survivor remains, promote the best one with a
    precise statement of what has been established and what remains open;
  - if none remains, write the scoped negative exactly in terms of `C` and `E`;
  - separately list any canonically representable but non-intrinsic objects as
    bookkeeping tools rather than mission-level wins.
- Expected output:
  - either a promoted intrinsic object memo,
  - or a scoped negative memo,
  - plus one short boundary statement explaining what the branch does not rule
    out outside `C`.
- Kill condition:
  - if the final memo overstates its scope beyond candidate class `C` and
    equivalence test `E`, the chain fails.

## Why this chain is meaningfully different

This branch still serves as the mission's intrinsicity guardrail, but it now
does so honestly. Its distinctive output is not a dramatic universal negative.
It is a disciplined scoped audit that can either promote a truly intrinsic
object or close a clearly defined class without confusing bookkeeping
convenience, dynamic tractability, and genuine intrinsicity.
