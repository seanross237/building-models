# Refined Chain 01 - Staged Joint-Gate Interaction Test On The Canonical One-Bridge Ledger

## Central Premise

The joint `G_tmpl` / `G_leak` question is real, but Step 6 does not yet
support an immediate full-frontier program on the entire canonical one-bridge
class.

The honest synthesis branch is therefore:

1. reconcile the controlling repaired ledger,
2. close the two exact object-local ledgers that Step 6 explicitly handed off,
3. and only then ask the smallest joint question that those closures actually
   support.

## Verifiable Result Target

Either:

- an exact gate-pair interaction memo on the carried record:
  explicit separation witness,
  co-orientation-only statement,
  or one partial implication under clearly stated extra hypotheses; or
- a restricted joint-frontier statement on one explicitly parameterized
  subclass with a declared ordering rule; or
- a bounded negative memo proving that the joint question is not yet
  theorem-ready until named exact ledgers are closed.

## Why This Chain Is Meaningfully Different

This chain still uses both promoted Step-6 objects together, but it does so
only after the smaller exact prerequisites are met. It is not just a weaker
copy of Chain 01 or Chain 02, and it does not pretend that shared ambient
packet class alone already yields a frontier theorem.

## Ordered Steps

### Step 1 - Reconcile The Controlling Record And Freeze Authority

Depends on: none.

Action: fix the authoritative gate sheet and traceability sheet for all joint
work. Use the Step-6 results as the controlling handoff, but explicitly log
the internal `L_cross` drift between
`step-005/explorations/exploration-001/REPORT.md`
and
`step-005/RESULTS.md` /
`step-006/RESULTS.md`.

Action: declare any ordering rule that a later
"strongest survivor"
claim would have to use, or else state that no such ranking may be used unless
an explicit subclass frontier is later closed.

Expected output: one authority memo fixing the live gate definitions,
hostile-data caveats, and survivor-order requirements.

Kill condition: if the record cannot name one controlling repaired sheet
without interpretive patchwork, stop and package a record-stability failure
memo rather than a synthetic joint result.

### Step 2 - Close The Two Step-6 Exact Ledgers On Their Smallest Carried Objects

Depends on: Step 1.

Action: for `G_tmpl`, close the exact `F_SL(rho)` ledger on
`0 < rho <= 1/16`
well enough to determine
`Delta_tmpl(rho)`,
`Delta_spec(rho)`,
and the true friendly extremal parameter.

Action: for `G_leak`, close the carried friendly-stress-set ledger on
`{F_SS(1/12), F_SL(1/16)}`
well enough to verify the repaired classwise vector and distinguish genuinely
sharp coordinates from merely retained ones.

Expected output: one exact or rigorously checked template-family ledger and one
exact or rigorously checked leakage-stress-set ledger.

Kill condition: if either object-local ledger still requires new packet
semantics, a new spectator partition, or invented hostile numerics, stop and
package the exact missing debt explicitly.

### Step 3 - Build A Shared Comparison Sheet Only Where Exact Data Now Exists

Depends on: Step 2.

Action: write both gates in one shared variable or packet-coordinate system,
but only on the now-closed carried record or on one explicitly parameterized
subclass inherited from those closures.

Action: mark each entry as exact,
symbolic,
or unavailable.
Use hostile data only to the extent that the same-currency record actually
supports it.

Expected output: one joint comparison table that separates friendly
admissibility, hostile exclusion, and unresolved entries without pretending to
globality.

Kill condition: if the shared sheet is still mostly placeholders or mixes
incompatible currencies, stop and report that no honest joint analysis is yet
available.

### Step 4 - Decide The Smallest Earned Joint Claim

Depends on: Step 3.

Action: test only the claims that the shared sheet can really support:
an explicit separation witness,
co-orientation only on the carried record,
or one partial implication under additional exact hypotheses.

Action: do not use equivalence or redundancy language unless a genuine
implication proof lands on disk. If no such relation is proved, say so
explicitly.

Expected output: one exact interaction memo for the gate pair on the closed
carried record or named subclass.

Kill condition: if the output is only suggestive language such as
"they seem related,"
the chain fails.

### Step 5 - Promote To A Restricted Frontier Only If The Prior Steps Earn It

Depends on: Step 4.

Action: attempt frontier packaging only if Steps 1 through 4 have already
produced both an exact shared sheet and a declared optimization order. Any
frontier claim must live on an explicitly parameterized subclass, not on the
whole one-bridge class by default.

Action: otherwise package the strongest honest endpoint as either a
separation/co-orientation memo or a bounded deferral memo naming the exact
missing ledgers.

Expected output: a restricted frontier theorem candidate, a clean
separation/co-orientation memo, or a theorem-readiness failure memo.

Kill condition: if the final endpoint depends on an undefined
"strongest" order,
an open-ended parameter sweep,
or a silently enlarged packet class,
stop and report failure.
