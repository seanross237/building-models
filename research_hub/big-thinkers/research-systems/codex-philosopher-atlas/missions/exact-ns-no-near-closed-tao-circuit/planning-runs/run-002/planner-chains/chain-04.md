# Chain 04 - Moonshot Near-Circuit Promotion From `F_SS`

## Central Premise

The portfolio still needs a real kill route for the firewall idea. The best
candidate on disk is not a hypothetical new family. It is the already recorded
positive witness
`F_SS(mu)`,
which is currently the only pro-circuit family that passed all three Step-4
candidates on one explicit exact sheet.

The moonshot is to promote that witness from a dossier pass to an actual exact
near-circuit candidate, using honest reduced dynamics first and allowing only
one minimal exact extension if the frozen witness nearly works.

## Verifiable Result Target

Either:

- a concrete exact packet or one-step exact extension that passes the promoted
  gate pair and exhibits a defensible Tao-like stage order strongly enough to
  count as a near-circuit counterexample candidate; or
- a constructive failure atlas showing exactly why the best positive witness on
  disk still stalls, leaks, or mistimes under honest exact dynamics.

## Why This Chain Is Meaningfully Different

This chain is constructive and dynamical. It does not start by proving an
obstruction theorem. It tries to beat the obstruction with the strongest
existing exact witness and only then records failure if the witness collapses.

## Ordered Steps

### Step 1 - Freeze The Constructive Target And Honesty Rules

Depends on: none.

Action: freeze
`F_SS`
as the base constructive family, with its frozen sign/phase sheets,
normalization,
mandatory conjugates,
and closure-forced companions.

Action: freeze the promotion scorecard in advance:
the two promoted gates
`G_tmpl`
and
`G_leak`,
plus the recorded stage-order diagnostics
`t_clk`,
`t_trig`,
`t_rot`,
`t_next`
from the earlier itinerary work.

Expected output: one constructive target sheet defining exactly what counts as
an honest near-circuit promotion.

Kill condition: if the positive witness only exists by silently retuning sign
or phase choices after the fact, stop and demote the branch immediately.

### Step 2 - Derive The Exact Reduced Dynamics Of The Frozen Witness

Depends on: Step 1.

Action: write the exact reduced amplitude system for the frozen
`F_SS`
packet, including all mandatory conjugates and closure-forced companions.

Action: make the coefficient ledger explicit enough that later dynamic success
or failure is attributable to named exact terms rather than scorecard drift.

Expected output: an explicit reduced ODE system with fixed initial data and
observable readouts.

Kill condition: if honest exact bookkeeping immediately forces a much larger
packet than the frozen witness allows, record that as constructive non-isolable
failure and stop.

### Step 3 - Test Whether The Frozen Witness Promotes Honestly

Depends on: Step 2.

Action: analyze or integrate the reduced system on the frozen window
`I = [0, 1]`
and test whether the witness keeps both promoted gate values admissible while
showing the intended stage order strongly enough to count as a Tao-like
near-circuit candidate.

Action: separate static-gate success from true dynamical success. A packet that
only passes static ledgers but fails the time-order diagnostics does not count
as a constructive kill.

Expected output: a promotion memo for the frozen witness or a named failure
mechanism on the exact reduced system.

Kill condition: if the witness fails cleanly at this stage, do not rescue it by
opening a broad parameter sweep.

### Step 4 - Allow One Minimal Exact Extension If And Only If The Witness Nearly Works

Depends on: Step 3.

Action: if the frozen witness nearly works but misses by one identified exact
bottleneck, allow one predeclared minimal extension such as one extra bridge,
one extra packet companion, or one fixed shell-locked mode.

Action: require full exact bookkeeping and re-evaluate the same frozen
scorecard on the extended packet. Do not allow iterative patching or a family
search disguised as one repair.

Expected output: either one strongest exact extended candidate or a proof that
even the minimal honest repair does not survive.

Kill condition: if success requires more than one exact extension, hidden mode
suppression, or post hoc threshold changes, terminate the branch as a failed
construction route.

### Step 5 - Package The Moonshot Verdict

Depends on: Step 4.

Action: package one of two outcomes:
an explicit exact near-circuit candidate that genuinely challenges the
firewall idea,
or a constructive failure atlas naming the decisive collapse mode:
timing failure,
spectator takeover,
coefficient misordering,
or forced packet growth.

Action: state exactly whether the verdict applies only to the frozen witness,
to the one repaired extension, or to a slightly broader exact class genuinely
earned in the branch.

Expected output: a counterexample candidate memo or a high-value constructive
negative memo.

Kill condition: if the branch ends with only
"search did not find a survivor,"
without one explicit candidate or one explicit collapse mechanism, it failed.
