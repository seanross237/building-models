# Attack on Chain 01 - Admissibility Versus Exact-Closure Separation

## Bottom line

This chain is attacking a real bottleneck, but it is too eager to upgrade an
epistemic gap into a mathematical separation theorem. The repository already
records three damaging facts: Step 6 froze `G_tmpl` and `G_leak` as screening
objects rather than implication theorems, Step 7 froze downstream work to the
single witness `F_SS(1/12)` on one canonical ledger, and Step 8 stopped at
exact non-isolability because no finite exact support-level closure ledger is
actually frozen. Against that background, much of this chain is not discovery.
It is repackaging.

The likely honest output is narrow: the current record does not justify any
implication from repaired admissibility to finite exact closure on the frozen
ledger, so `G_tmpl` and `G_leak` should remain screening tools. That is a
useful memo. It is materially weaker than the chain's more ambitious rhetoric
about proving separation or identifying a minimal repair.

One immediate hygiene problem also matters: the selected file path is
`chain-01.md`, but the document header says `# Chain 02`. That is not fatal to
the mathematics, but it is sloppy in a branch whose whole credibility depends
on freeze discipline.

## What is genuinely strong

- It correctly identifies that exact closure, not more static screening, is the
  real mission bottleneck.
- It keeps insisting on same-ledger comparison instead of allowing currency
  drift.
- It leaves room for a negative memo instead of pretending the repaired gates
  are already closure-level objects.

## Step-by-step critique

### Step 1 - Freeze The Implication Sheet To Be Tested

This step is weaker than it looks because most of its content is already on
disk.

- Step 6 already froze the promoted gates without an exact implication between
  them.
- Step 7 already froze the downstream authority sheet and same-currency rule on
  the canonical packet ledger.
- Step 8 already recorded that the branch lacks one finite exact closure ledger
  for `F_SS(1/12)`.

So Step 1 is not really isolating a new research object. It is mostly restating
the current dossier.

The more serious problem is quantifier ambiguity. "Does `G_tmpl` imply finite
closure" can mean at least four different things:

- for the single frozen witness `F_SS(1/12)`,
- for every packet on the canonical one-bridge ledger,
- for every packet that passes the repaired thresholds,
- or in the much weaker record-relative sense "can the current repository
  derive closure from this gate."

Those are not interchangeable. The chain never freezes which one it means, so
the rest of the plan can drift between witness-local failure and theorem-level
non-implication.

Kill-condition problem:

- The stated kill condition is far too weak. The branch should also stop if it
  cannot freeze the quantifier domain and the exact meaning of "implies."
- It should also stop if the implication sheet is already effectively answered
  by existing Step-6 to Step-8 notes and only needs packaging.

### Step 2 - State One Honest Exact-Closure Predicate On The Same Ledger

This is the chain's real choke point, and the current record already says it is
not solved.

- The repository explicitly says it does not yet freeze one support-level exact
  closure realization for `F_SS(1/12)`: no explicit wavevector family, no
  fixed helicity assignment, no exact coefficient ledger, and no finite closure
  convention that lists which forced companions must remain.
- Because of that, Step 8 stopped at a raw classwise closure-burden ledger
  rather than a finite exact subsystem.

That means Step 2 is not a normal definition step. It is asking the branch to
repair exactly the thing the record says is still unfrozen.

The example predicate is also suspect. "Existence of one finite exact closed
subsystem" sounds innocent, but in this context it can smuggle in a new support
choice, a new truncation rule, or a new closure convention. If the branch gets
to choose one existential witness after the fact, it has already relaxed the
freeze discipline that Step 8 was defending.

Kill-condition problem:

- The kill condition should not just reject support-semantic changes and
  arbitrary truncation. It should also reject vacuous predicates that are true
  only because the branch weakens closure until it becomes trivial.
- If Step 2 fails, that is not a mid-chain setback. It already settles most of
  the chain's downstream ambition: the implication question is not theorem-ready
  because the target object is not frozen.

### Step 3 - Populate The Relation Matrix With The Carried Witnesses

This step breaks the branch's own freeze discipline.

- Step 7 froze the downstream witness to the single packet `F_SS(1/12)`.
- Step 3 now wants to populate the matrix with `F_SS(1/12)`, `F_SL(1/16)`, and
  `F_DT(delta, eta)`.

That is witness-set drift. `F_SL(1/16)` and especially the hostile family
`F_DT(delta, eta)` may still be relevant historically, but they are not the
same thing as the single frozen witness carried into Step 8. Reintroducing them
here quietly changes the scope of the branch.

The matrix also mixes unlike statuses:

- gate pass or fail on repaired Step-6 screens,
- closure status on the Step-8 frozen ledger,
- and family-level versus witness-level objects.

Those do not sit in one clean table unless the chain first re-derives them in a
single exact domain. It has not done that.

The biggest overreach is using `F_SS(1/12)` as a "separation witness." Step 8
showed that the current record does not yet freeze one finite exact closed
subsystem on the same ledger. That is not automatically the same as proving
that `F_SS(1/12)` falsifies every honest finite-closure predicate, and it is
certainly not yet a theorem that gate pass does not imply closure in the
underlying mathematical sense.

Kill-condition problem:

- The stated kill condition watches for packet-semantics drift, which is good.
- But it misses the more immediate failure: the branch has already drifted from
  the Step-7 single-witness freeze just by reopening `F_SL` and `F_DT`.

### Step 4 - Prove The Strongest Honest Separation Or Minimal Repair

This step offers the wrong menu of honest outcomes.

- One honest outcome is not a separation theorem or a repaired implication.
  It is simply: "the closure object is still unfrozen, so no implication claim
  is theorem-ready."
- The chain treats that possibility mostly as a failure mode on the way to
  stronger packaging, but it may actually be the main result.

The "minimal repair" branch is especially dangerous. Without a strict
anti-tautology rule, it invites exactly the wrong move:

- add a support-level closure hypothesis,
- add existence of the finite closed subsystem as an assumption,
- or add some bespoke exact companion rule that is just closure in disguise.

That would make the implication true by smuggling the target into the premise.
Calling that a repair would be misleading.

The proposed example
"joint repaired admissibility does not imply finite honest closure"
is also too strong for the current evidence base. What the record supports is
roughly this:

- repaired admissibility has not been shown to yield finite exact closure on the
  frozen ledger,
- and the earliest constructive obstruction currently on disk is exact
  non-isolability / arbitrary truncation.

That is weaker than a clean non-implication theorem unless the branch exhibits
an explicit same-ledger counterexample or proves that every possible closure
realization fails.

Kill-condition problem:

- Rejecting vague "moral relatedness" is correct.
- But the chain also needs to reject fake repairs that merely restate closure as
  an extra hypothesis under a new name.

### Step 5 - Decide The Downstream Consequence For Mission Objects

This step is mostly predetermined by the earlier record.

- If Step 8 already says no finite exact closure object is frozen, then the
  default downstream consequence is already plain: `G_tmpl` and `G_leak` remain
  screening tools, not closure theorems.
- The alternative "promote a third closure-level object" is premature unless
  the branch has actually frozen such an object. Right now the repo mostly
  records the opposite.

So Step 5 risks producing bureaucratic motion rather than substantive progress.
It can easily end in "name the missing closure object now required," which is
just a more ceremonious way of restating the Step-8 stop verdict.

Kill-condition problem:

- The final kill condition guards against reopening broad definition work too
  early, which is sensible.
- But it does not guard against the more likely failure: writing a polished
  downstream memo whose content is already forced by the existing Step-8 record.

## Structural weaknesses of the whole chain

- Prior-art overlap inside the repo is high. Step-6 to Step-8 factual notes
  already say the promoted gates are screening objects, later comparisons must
  stay on one frozen ledger, and the current record stops at exact
  non-isolability rather than finite exact closure.
- The chain also overlaps earlier planner work on relation mapping between
  `G_tmpl` and `G_leak`. Its claimed distinctiveness is only partly real.
- Quantifier drift is the central technical flaw. The chain slides between
  witness-local claims about `F_SS(1/12)`, relation claims on the frozen ledger,
  and theorem-level implication claims about the promoted gates.
- It mistakes an epistemic deficiency for a mathematical separation. "No finite
  closure ledger is frozen" is not automatically the same as "closure is
  separated from repaired admissibility as a theorem."
- The chain is biased toward a predetermined conclusion. Its endpoint menu is
  basically:
  separation theorem,
  repaired implication,
  or promotion of a third object.
  There is too little room for the most mundane honest result: the current
  dossier still only supports demotion to screening status.
- The kill conditions are miscalibrated. They are strict about rhetoric and
  bookkeeping cheating, but too weak about repackaging known results or letting
  scope drift away from the Step-7 freeze.
- Administrative sloppiness matters here. A selected `chain-01.md` whose title
  says `Chain 02` is a small but telling signal that the branch has not fully
  internalized its own freeze discipline.

## Fair verdict

There is a real idea here: the mission should not keep pretending that repaired
static admissibility is the same thing as finite exact closure. But as written,
this chain overshoots what the current record can honestly support.

The strongest fair endpoint is probably a short memo, not a five-step branch:
on the frozen ledger, the repository still has no finite exact closure object;
therefore neither `G_tmpl` nor `G_leak`, alone or jointly, should be treated as
implying honest finite closure. Keep them as screening tools only. Do not call
that a separation theorem unless the branch freezes an explicit same-ledger
counterexample rather than merely inheriting Step-8's missing-data verdict.
