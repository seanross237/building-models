# Separate True Symmetries Canonicalization And Substantive Object Changes Before Claiming Support Stability

When `support` is mission-central, freeze the smallest object that keeps exact
bookkeeping honest before making any stability claim. In packet branches, that
often means promoting packet identity above single-mode bookkeeping and treating
conjugate completion as mandatory canonical representation rather than as the
primary object itself.

Then split representation moves into three classes:

- true symmetries of the chosen object
- canonicalization choices fixed once up front
- substantive object changes that create a different model

Until that split is frozen, packet regrouping, helical-basis changes, and even
conjugate-completion language should not be assumed cosmetic. This is a common
failure mode behind false claims that an object is "stable up to
representation."

Once a branch has already frozen one canonical packet sheet, later robustness
work should ask a narrower question: does the candidate survive true exact
symmetries, does it merely depend on the declared honest ledger, or has the
packet model changed? Otherwise the branch can mistake `depends on the chosen
sheet` for `mathematically nonexistent`.

On the same logic, sign-sheet, threshold, spectator-partition, and time-window
changes cannot be reclassified as canonicalization after outcomes are already
inspected. If those items move after the freeze, the branch has changed the
object it is auditing.

The same caution applies inside a frozen sign-sheet catalog. Mixed sign classes
should collapse only under equivalences that the record actually froze.
If the sheet justifies conjugation, one global helical-sign relabeling, and an
internal role swap, that does not yet license any stronger target/input
permutation collapse between distinct mixed-sign classes. Over-collapsing those
classes hides real support distinctions and makes the later catalog look more
canonical than the frozen object sheet supports.
