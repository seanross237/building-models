# Critique: Proof Architecture / Theorem-Object Stability

## Strongest Objections

### 1. The draft makes the least stabilized route the main route

The authoritative status file does **not** support elevating the critical-element
ledger bridge to the main program. It says that route is only conditionally
alive and is blocked at the canonical-ledger extraction lemma. In project terms
it is explicitly parked behind the packet-screen theorem work and the intrinsic
phase route. The draft reverses that ranking without earning the reversal.

From a proof-architecture standpoint this is the central error: it promotes the
hardest and least formalized object-definition burden to the front of the plan,
even though the repository record says that burden is earlier, deeper, and less
stable than the downstream screen theorems.

### 2. The main target is not actually a frozen theorem target

`Exclude a symmetry-normalized minimal blowup element in an accepted critical
host space, defaulting to L^3` is not a theorem target. It is a slogan with
several unfrozen choices:

- ambient domain: `R^3` or `T^3`
- solution category: mild / Leray-Hopf / suitable / almost periodic modulo symmetries
- exact critical space: `L^3`, `\dot H^{1/2}`, or something else
- symmetry group and normalization
- exact blowup scenario being excluded

These are not cosmetic choices. They determine what compactness technology is
available, what extraction statement even means, and whether the downstream
helical/triad object is compatible with the PDE setting.

Right now the draft hides all of that inside “accepted critical host space” and
“symmetry-normalized.” That is underdefined at the level where the proof would
actually break.

### 3. There is a domain/object mismatch that the draft does not confront

The packet/ledger frontier in the status file is built on exact helical triad
objects and one-bridge ledgers. The compactness-rigidity target named in the
draft defaults to `L^3` minimal elements, which is a very different proof
environment. The draft never states the exact map from a minimal blowup object
in the chosen PDE category to the helical one-bridge ledger category.

That is not a missing detail. That map is the route.

Until the plan freezes:

- the ambient PDE setting,
- the object category,
- the topology/equivalence relation on extracted objects,
- and an equivariant extraction statement,

the contradiction chain is not an architecture. It is a wish list.

### 4. Phase 0 is pretending the main theorem burden is “setup”

The draft’s “Freeze the object layer” phase includes:

- canonical one-bridge object
- comparison currency
- harmless refinement
- relabeling invariance
- reconstruction rule
- meaning of canonical extraction

That is not preparatory bookkeeping. That is the main mathematical problem.

Worse, the status file already says the current frontier stops because there is
no canonical finite reduction, no promoted same-currency explicit realization,
and no canonical extraction rule from exact NS data. So Phase 0 is not an
organizational step before theorem work. It is a cluster of unsolved theorem
obligations, some of which may fail outright.

### 5. The draft is trying to stabilize too many abstractions at once

The main route simultaneously asks for:

- a critical-element framework,
- a canonical extraction functor,
- a one-bridge object class,
- a stable comparison currency,
- quotient-invariant observables,
- and a contradiction theorem.

This is overbuilt. Prize-level proof programs fail when too many unstable
interfaces are being frozen simultaneously. A minimum theorem surface has to be
frozen first, and the plan does not identify that surface.

### 6. “Canonical extraction” is likely too strong unless formulated on a quotient

The draft asks for extraction stable under compactness limits, harmless
refinement, and relabeling. That is reasonable as a desideratum, but it is not
yet a mathematically coherent theorem statement. In concentration-compactness
settings, exact canonical representatives are often unavailable; what survives
is an extraction to an equivalence class, selector, or invariant quotient.

The draft never states:

- what counts as the same extracted object,
- whether the output is an object or an equivalence class,
- what topology lives on that class,
- or which observables descend to the quotient.

Without that, “canonical extraction lemma” is formal language, not a theorem.

### 7. The fallback route is still not insulated from the known structural kill

The fallback route names combined strain alignment and vorticity-direction
coherence, but the status file already records a hard obstruction for the
`e_2`-alignment line: `s_2 > 0` persists in high-vorticity regions and the
alignment mechanism itself removes the feedback that could reverse it. Unless
the fallback theorem is stated in a way that explicitly avoids this obstruction,
the plan is reopening a route that the status file says is structurally blocked.

## Missing Assumptions / Prerequisites

The draft should not branch further until it freezes the following.

1. The exact PDE setting.
   `R^3` vs periodic domain, solution class, blowup notion, and which accepted
   theorem target is being touched.

2. The exact critical-element framework.
   If the plan says `L^3`, it should name the compactness-rigidity framework it
   intends to inhabit and why the object-extraction problem is even compatible
   with that framework.

3. The extraction codomain.
   Is the output a ledger, a quotient class of ledgers, a transfer tensor on an
   intrinsic substrate, or something else?

4. The equivalence relation.
   “Harmless refinement” and “relabeling” cannot stay informal. They need a
   mathematically defined equivalence relation on the candidate object class.

5. Well-defined observables on the quotient.
   Template-defect and leakage only matter if they descend to the equivalence
   class and are stable in one defined currency.

6. The first theorem statement.
   Not a chain. One actual theorem statement with hypotheses, conclusion, and
   objects fully typed.

## Concrete Revisions

### Revision 1: Demote the current main route back to conditional status

Do not present the critical-element contradiction route as the main route.
Present it exactly as the status file supports it:

- conceptually attractive,
- conditionally alive,
- blocked at canonical-ledger extraction,
- not to be advanced unless the extraction problem is formalized first.

### Revision 2: Insert a pre-route feasibility gate

Before any contradiction program, require one of the following:

1. an intrinsic extraction theorem, or
2. a no-go theorem showing that no packet-ledger extraction with the desired
   invariances can exist in the chosen setting.

That is the correct first branch from a proof-architecture perspective. If the
answer is negative, the plan should stop pretending the contradiction route is a
live mainline.

### Revision 3: Freeze the minimum theorem surface, not the whole object layer

The minimum surface to freeze before any further branching is:

1. ambient PDE setting and target theorem,
2. candidate extraction codomain,
3. equivalence relation on outputs,
4. one quotient-invariant observable pair,
5. one exact theorem statement connecting PDE data to that quotient object.

Anything beyond that is premature.

### Revision 4: Prefer quotient-first formalization over representative-first language

Replace “canonical one-bridge object” with language like:

- intrinsic output class,
- quotient under refinement/relabeling,
- selector only if needed later.

This is a more realistic formal target. It separates theorem content from
representative choice.

### Revision 5: Make the fallback route a single theorem statement immediately

The fallback should be written as one precise conditional regularity statement
that explicitly says why it does not collapse to the blocked `e_2`-alignment
program. If that cannot be written now, the fallback is not mature enough to be
a fallback.

### Revision 6: Align the plan with the repository’s proven dependency graph

The status file’s theorem-facing near-term work is:

- promote the frozen template-defect screen into a real theorem, or
- promote the frozen leakage screen into a real theorem, or
- pursue the intrinsic shell-crossing phase route in its stricter recursive-spill form.

If the final plan wants a different ordering, it must explicitly justify why the
earlier dependency obstruction is no longer binding. Right now it does not.

## Salvageability

The plan is salvageable **without** a full rewrite, but only if it is cut down
hard and reordered.

What is salvageable:

- the insistence on accepted Navier-Stokes theorem targets,
- the stop-rule discipline,
- the refusal to count private artifact cleanup as progress.

What is not salvageable in its current form:

- treating the critical-element ledger contradiction as the main route,
- treating object-layer freezing as a prelim rather than the main theorem risk,
- leaving the PDE setting and extraction codomain underdefined.

So the right verdict is:

`salvageable after major re-scoping, not salvageable as currently prioritized.`
