# Scope and Simplicity Critique

This plan is overbuilt. It reads like five research programs stacked on top of each other, all using slightly different vocabulary for the same unresolved object-definition problem. That is not a focused path to a theorem. It is a hedge against uncertainty disguised as a plan.

## Strongest Objections

The plan tries to do too much at once:

- It pursues four substantive theorem tracks plus a sidecar search plus a conditional compactness-rigidity bridge. That is not a ranked plan; it is a research portfolio with no real pruning.
- It keeps renaming the same basic obstacle as if more terminology creates progress: packet-family theorem, leakage firewall lemma, intrinsic spill-versus-locking theorem, exact triads, canonical extraction lemma, no-double-admissibility. This is scope creep through rebranding.
- It demands exact canonicity, same-currency comparison, harmless-refinement stability, exact closure bookkeeping, and route-specific theorem roles before any route is allowed to breathe. Those are not small hygiene checks; they are a second research agenda.
- The plan treats ledger-specific formulas and frozen-sheet identities as if they are near-term theorem targets. That is a dangerous overfit. A plan should identify a general theorem shape, not prematurely commit to a brittle bookkeeping artifact.
- Route 5 is especially bloated: it adds a compactness-rigidity bridge on top of all earlier routes, but only if a canonical extraction lemma appears. That makes the plan both wide and conditional in a way that guarantees indefinite deferral.

The central problem is not just that the plan is ambitious. It is that the plan has too many moving parts before it has earned one stable abstraction. That makes it hard to know what success even means.

## Missing Assumptions or Prerequisites

The draft assumes several things it never justifies:

- That the frozen ledger is a durable theorem surface rather than a one-off artifact.
- That exact packet-family identities or leakage inequalities are actually connected to a real NS regularity mechanism, rather than just to a curated internal notation system.
- That a canonical object can be extracted at all before the plan knows what the canonical object should be.
- That the non-Tao sidecar directions are genuinely independent enough to be worth separate work, rather than being restatements of the same closed estimate architectures.
- That “prize-relevant progress” can be decomposed into route-local theorem objects without first proving a unifying bridge from those objects to regularity or blowup prevention.

The plan also quietly assumes that the object-definition frontier is the right frontier to attack first. That may be true, but the draft never shows why the complexity cost of building that object layer is lower than simply narrowing the question to one clean theorem target.

## Concrete Revisions

- Collapse Routes 1-3 into one main route and one fallback. Right now they are three variants of the same object-centric chase.
- Delete the capped sidecar program unless it can be reduced to one named lemma target. A “strictly capped search” is still a search program, and it is still scope expansion.
- Remove Route 5 from the main plan. Compactness-rigidity should not be in the plan at all until a single canonical object exists and has been shown to survive refinement.
- Replace the route-specific formula targets with one higher-level theorem target per route. The current targets are too ledger-bound and too easy to mistake for progress when they are only bookkeeping.
- Cut the proof standards down to the minimum necessary to preserve correctness. The current admissibility checklist is so broad that it becomes a second project.
- Reduce the success criteria to one of two outcomes: either a genuinely canonical theorem object, or a clean no-go result that closes the object class. Everything else should be treated as exploratory note-taking, not plan structure.

## Salvageability

The plan is salvageable without a full rewrite, but only if it is aggressively narrowed. The current version is too wide to execute cleanly, but the core idea of “start from the live object-definition frontier and demand exact theorem objects” can survive.

What is not salvageable is the current multi-route architecture. If this remains a five-route program with sidecars, conditional bridges, and route-specific bookkeeping targets, it will keep generating activity without producing a simpler theorem path. The plan needs one main route, one fallback, and one explicit stop condition. Anything more is scope inflation.
