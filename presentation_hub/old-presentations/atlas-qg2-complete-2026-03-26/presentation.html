The System

5 Agents, 3 Levels

Each agent is a Claude Code instance in its own tmux session. The hierarchy decides who owns which scope of decisions.

Missionary
Writes strategies. Evaluates results. Decides when to stop.
Strategizer
Owns one strategy. Designs explorations. Runs 8–16 per strategy.
Explorer
Researches one question deeply
Librarian
Searches the shared knowledge base
Curator
Files findings into the library
OUTER LOOP
Missionary writes a strategy (methodology + goals), launches a strategizer, reads the final report, decides whether to write another strategy or declare mission complete.
INNER LOOP
Strategizer queries the librarian, designs an exploration goal, launches an explorer in tmux, reads its report, launches the curator to file findings. Repeat until strategy is exhausted.
QG2 was the second full mission on this architecture. The missionary ran 4 strategies (vs. 2 in QG1), demonstrating the outer loop can sustain longer campaigns.

This is the main story. Each strategy's methodology was a direct response to the last one's failures.

S-001: Diverge → Construct → Stress
Good divergence phase, but premature convergence. Multi-task explorations timed out regularly. Built something creative that couldn't compute anything.
Lesson: multi-task explorations fail Lesson: need earlier adversarial testing
S-002: Gap-Mine → Sprint → Build → Harden
Best of the first two strategies. Parallel sprints were excellent. But interpretive targets don't produce testable theories — every sprint hit the classical-to-quantum gap.
Win: parallel sprints Lesson: interpretive targets are dead ends

Key insight
The system learned across strategies. Each strategy was designed by the missionary as a direct response to what failed before. This is exactly the behavior you want from the outer loop.

Takeaways

The Golden Rules

Six rules the system discovered about how to run autonomous research. Learned the hard way, across 36 explorations.

1
One task per exploration. Non-negotiable. Multi-task explorations time out, produce shallow results, and waste context. Every exploration gets exactly one job.
2
Devil's advocate immediately after first construction. Don't build for 5 explorations then test. Build once, stress-test once, then iterate. Catches fatal flaws before you invest.
3
Pre-load ALL relevant context. Include specific paper names, equation numbers, and prior results in the exploration goal. Explorers that start cold produce shallow work.
4
Every exploration gets a verdict. Not just a report — a classification. Supported, refuted, inconclusive. Forces the explorer to commit to a conclusion.
5
200–400 lines per focused exploration. Longer than this means the task was too broad. Shorter means the explorer didn't go deep enough.
6
“If this fails, explain WHY” in every goal. Without this instruction, explorers paper over failures. With it, failed explorations are as informative as successful ones.
These rules are domain-agnostic. They apply to any research mission run on the Atlas architecture, not just quantum gravity.

Takeaways

The Boundary

The mission ended because the system discovered what AI text agents can and cannot do for research.

What text agents CAN do

Survey and synthesize. Map an entire field's landscape, find gaps, identify connections between theories.
Build frameworks. Construct logically consistent theoretical structures from existing pieces.
Stress-test ruthlessly. Devil's advocate mode finds real flaws. Three serious attacks, zero fatal — that's useful validation.
Know when to stop. The system identified its own limit and specified exactly what computation was needed next.

What text agents CANNOT do

Produce discriminating predictions. Strategy 4's devil's advocate wiped out every prediction. Reasoning about formulas is not computing them.
Run numerical calculations. The three critical computations require FRG flow equation solvers. No amount of text reasoning substitutes for this.
Cross the classical-to-quantum gap. Every concept sprint in S-002 hit this wall. Bridging it requires new mathematical machinery, not argument.
Bottom line
The program reached the boundary of what can be achieved by reasoning about existing theories without performing original calculations. The three critical computations it identified require numerical FRG expertise in human research groups. The system didn't just fail and stop — it diagnosed exactly why it couldn't go further and specified exactly what would be needed to continue. That's the product.
