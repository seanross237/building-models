Head Chef
Bonsai

Core bet
Don't optimize 20 knobs independently. Bundle them into recipes and optimize the recipe.
Why simpler
One decision per node (which recipe?) instead of a matrix of independent variable choices.
Still flexible
Recipes evolve via mutation. Fork a recipe, change 1-2 fields, benchmark it, keep the winner.

Anatomy

What is a node?

A directory on disk. Five files, five concerns. Nothing hidden.

node-042/
input.json — task + instructions
recipe.json — frozen config snapshot
output.json — result (exists = done)
children/ — child node dirs (if any)
trace.json — tokens, timing, cost
Identical DNA
Every node has the same structure. Pull any node from the middle of a tree — same shape as the root.
No status machine
output.json exists? Done. Doesn't exist? Not done. Recovery = re-run the node.
Fully inspectable
Open any node's folder, read its files, understand exactly what happened. No log archaeology.

Lifecycle

How does a node actually run?

Receive task + recipe. Call model once. Three possible outcomes.

📥
Receive
Task + recipe arrive from parent (or mission root)
🔧
Expand
Recipe values map to prompt templates, tool policies, model config
🧠
Call
Single model call with expanded prompt
📤
Resolve
One of three outcomes (below)
✅
Answer
Write output.json, done.
🌳
Decompose
Spawn child nodes with subtasks. Wait. Synthesize.
🔍
Retrieve
Fetch info, then re-prompt with new context.
The orchestrator walks the tree depth-first. No scheduler, no queue. Just recursion.

Bonsai

How does the system learn?

Three levels of optimization, each building on the last.

Level 1 Best Average Recipe
Run every recipe through the benchmark suite. Pick the one with the highest average score. Use it as default. A bash script.
Level 2 Best Recipe Per Category
Simple heuristic lookup: "for math, use methodical-v3; for creative, use exploratory-v2." A JSON file.
Level 3 ML Recipe Selector
A model takes task features and outputs a recipe name. Trained on the scoreboard. Later.
Recipe Evolution
Take the best recipe. Change 1-2 fields. Name it as a new version. Run benchmarks. Keep it if it wins. This is the Rhyme-Eval mutation rate insight applied to system config.

Design Rationale

Why this shape and not another?

Recipes (this design)
  • One decision per node: which recipe?
  • Human-readable names, easy to discuss
  • Fork + mutate = natural evolution
  • Scoreboard row is simple and flat
  • Level 1 optimization is a bash script
vs
Independent variables
  • N decisions per node: one per knob
  • Combinatorial explosion (10 fields x 5 values = 10M combos)
  • Interactions between variables are invisible
  • Need multi-dimensional optimization from day 1
  • Hard to name or discuss a configuration
~600
Lines of Python (est.)
5
Files per node
1
Decision per node
0
Databases needed
The simplest system that could work: a node runner, a recipe expander, a benchmark harness, and a flat scoreboard file. If it needs to get more complex later, the data format already supports it.
