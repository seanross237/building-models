What matters most

This first pass proved the bones: live execution, storage, grading, and replay are all real. The system is not fake scaffolding anymore.

What still limits it

The question bank is only partly runnable, and the main mode’s branching adds trace depth faster than it adds accuracy on the hard cases.

Story arc

Build the runtime, wire live provider execution, store every run cleanly, then compare baseline and main on a small explicit batch. The result: one clean single-node win, one clean delegated win, and two delegated failures that are still informative.

Execution path

How does a run proceed?

1
Task enters `run_v1.py`
The runtime reads the task text plus run-level settings like provider, model, and depth cap.
2
Run packet and defaults are set
The run packet becomes the canonical input, with a predictable local path still available.
3
Root node starts the work
The root node receives a prompt snapshot and begins local solving or synthesis.
4
Heuristic branching may recruit helpers
Baseline stays at `max_depth=0`. Main mode can branch up to depth 3.
5
Each node writes replay and records
Node packet, node record, events, edges, prompt snapshot, raw model payload, and tool transcript are all captured.
6
Run summary and derived views are emitted
A calm end-state lands in `run_summary.json`, `derived/timeline.json`, and `derived/timeline.md`.
Where does branching actually matter?

On the live batch, single-node runs stayed clean on the easy exact-answer case, while delegated runs only sometimes improved anything. Structural delegation is real; quality improvement is still inconsistent.

Bottom line

The live provider seam is real. The system now proves it can execute and store live runs. The next challenge is to make delegation improve outcomes, not just branch the tree.

Implementation choices

What judgment calls were made?

Why start with a deterministic executor?

It made v1 runnable and testable immediately while preserving the live provider seam for later.

Why choose file-first storage?

Canonical truth under `data-system/run-history/` stays simple, reconstructable, and aligned with the target docs.

Why keep multi-step root history calm?

Replay can show the authored steps, while the canonical node record stays readable and gets a `final_action_type`.

Why hard-cap depth at 3?

That was the requested stop against runaway branching, and it belongs in code rather than convention.

Why begin correctness with stored-run validation?

The first honest step was to prove the written run matches the contracts and layout before pretending dry-run machinery exists.

Why use a small explicit grading subset first?

The bank is only partly runnable, so a narrow honest grader was the fastest path to a real baseline-vs-main batch.

Open questions and next moves

What remains open?

Open questions
Should delegation be driven by prompting, explicit variables, model judgment, or a mix?
Should the grading-bank files become richer runnable packets instead of summary markdown cases?
Should multi-step node histories stay hidden in replay or become more formal in core records?
How far should correctness grow beyond stored-run layout validation?
Pragmatic recommendations
Make runnable question packets richer and decomposition decisions more explicit.
Add stronger verification for exact-answer questions and separate synthesis prompts from local-solve prompts.
Prefer exact-answer numeric/string cases and keep the same cheap reliable model for short integration loops.
Add question-level metadata for `recommended_baseline_mode`, `recommended_main_mode`, and `recommended_grader`.
What Sean should leave with

The system is no longer aspirational. It is a live, stored, reviewable first pass with a real batch result. The main gap is not execution, it is making delegation reliably earn its extra complexity.
