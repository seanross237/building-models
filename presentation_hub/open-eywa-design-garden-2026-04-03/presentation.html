TL;DR

What is the punchline?

This proposal splits the system so each layer has one job and one trust boundary.

🌳

Eywa solves tasks

It runs the tree, records every node, and keeps the execution contract stable.

🌿

Bonsai tunes variables

It learns from run history and benchmarks to choose better settings for each task shape.

🏺

Potter changes code only

It accepts a refactor only when the dry run machine keeps outputs identical.

Execution tree

What is Eywa?

Eywa is the stable node machine. A root task becomes a tree, but every node keeps the same DNA.

instruction packet
tree variables
node variables
root task
Only two things vary: the instruction packet and the variable bundle.
Everything else stays fixed: lifecycle, artifacts, events, usage, and terminal outcomes.
Why this matters: any middle node should still look like the same machine.

Contracts

What stays fixed?

The system only works if the hard side stays hard.

Node DNAstable unit
  • same lifecycle across all nodes
  • same artifacts for a given role
  • same event meaning across runs
Variable bundleonly flex point
  • tree scoped defaults
  • node scoped overrides
  • closed family set not ad hoc knobs
Raw factsappend only
  • events before summaries
  • usage and cost recorded per run
  • history never rewritten silently
Potter gateno behavior drift
  • dry runs must match
  • live outputs must stay equivalent
  • timing drift only within tolerance
Open questions

What is still open?

These are the decisions that still need Sean to choose the shape of the roots.

1

Which variables are canonical?

The system needs a closed set before it can learn cleanly from runs.

2

How much canonicalization is enough?

Task text may need a compact form before embeddings or lookup can work well.

3

What counts as equivalence for Potter?

Some outputs must match exactly. Some timings may only need to stay within tolerance.

The first version should bias toward strict contracts, heuristics, and inspectable history.
