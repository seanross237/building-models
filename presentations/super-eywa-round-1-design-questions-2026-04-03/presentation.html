Decision 1

What is the real optimization surface?

This decides whether Bonsai is learning named bundles, explicit knobs, or both.

Explicit Variables
Most legible
The truth is a resolved variable schema. Every run records the exact knobs that shaped it.
Best for clean learning and comparison. Harder to keep human-friendly.
Recipes
Most compact
The truth is a named recipe or profile. Bonsai chooses one object instead of many knobs.
Best for simplicity. Risks hiding behavior inside bundles.
Hybrid
Most balanced
Recipes exist as presets, but the resolved variable bundle is the source of truth and what gets recorded.
Best if you want both operator simplicity and long-term learnability.
Why does this matter?
This choice affects how Bonsai learns, how runs get compared, how humans inspect the system, and how much hidden behavior can sneak into the machine. If this is fuzzy, the later target docs will be fuzzy too.
Decision 2

What must every node always expose?

This is the base contract. Everything later depends on what a node must contain and emit.

Minimal Folder
Leanest
Input, resolved vars, output, metrics, children. Nothing more is canonical at first.
Fastest to build. May be too thin for later learning and replay.
Spec + Trace
Best contract
Every node has a clear spec plus a canonical trace packet that says what happened.
Strong boundary for Eywa, Bonsai, and later Potter.
Rich Artifact Set
Most complete
Prompt refs, tool logs, lineage, scoring hooks, replay signatures, and more are part of the contract.
Most future-proof. Highest risk of freezing too much too early.
Node sameness depends on this being explicit
Decision 3

What actions may a node take?

This defines Eywa’s behavioral surface. A small action vocabulary is easier to reason about, but may feel constraining.

Tight Core
Most sturdy
Solve, split, delegate, escalate. Everything else is a sub-shape of those actions.
Very clean. Keeps protocol small.
Planner Runtime
Middle ground
Solve directly, plan and solve, plan and delegate, plan and split.
Matches how you naturally talk about the system today.
Broad Action Set
Most expressive
Retrieve, retry, self-review, synthesize, verify, and more all become first-class actions.
May overfit early implementation detail into the core contract.
What does this unlock?
Once the action vocabulary is frozen, it becomes much easier to define the node lifecycle, the trace packet, and what counts as a comparable run.
Decision 4

Where should specialization live?

This is the question of tree defaults, node overrides, and how much freedom a node has to differ from its siblings.

Tree Defaults Only
Most uniform
One chosen bundle for the whole run. Nodes do not get their own specialization.
Cleanest for comparison. Weakest for hard mixed tasks.
Parent-Set Overrides
Most practical
The tree sets defaults, and parents may assign explicit node overrides when a child needs a different shape.
Keeps control inspectable while allowing local adaptation.
Dynamic Self-Rewrite
Most adaptive
Nodes can rewrite or request their own variables mid-flight as they learn more about the task.
Powerful, but much easier to hide behavior and muddle the learning surface.

Node independence
Affects

Risk

Decision 5

What data should every run emit?

This decides what Bonsai can learn from and what future replay or equivalence checks can depend on.

Scoreboard Row
Smallest
Task tags, chosen config, score, cost, time, node count. One clean row per run.
Great for heuristics. Thin for deep debugging or replay.
Trace + Score
Best base
A canonical trace packet per node plus a scored run record per completed run.
Good balance between learnability and inspectability.
Full Replay Artifact
Most ambitious
Keep enough raw material that future replay and equivalence checks can reconstruct behavior exactly.
Very powerful. Easy to overbuild before the core loop is clear.
Why is this a round-one question?
Because if the first runs do not emit the right facts, later learning and later correctness checking will both be weaker. This is one of the few places where over-simplifying early can cost a lot later.
Decision 6

How should tasks be represented before ML?

This is the bridge between a messy real-world task and something Bonsai can actually use for retrieval and comparison.

Hand Tags Only
Simplest
Use a small stable taxonomy like domain, task shape, verification mode, ambiguity, and decomposition need.
Very interpretable. May throw away too much signal.
Tags + Canonical Form
Best bridge
Keep hand tags, plus an LLM-written normalized task form that strips noise before storage and lookup.
Keeps human semantics while making later ML easier.
Embeddings Early
Most ambitious
Generate embeddings from the raw or normalized task from the start and use them for similarity search.
Powerful later. May be premature before the basic contracts are stable.
After this round, the next deck should move from these choices to detailed schema questions.
