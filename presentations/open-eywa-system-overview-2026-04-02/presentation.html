Node JSON

The node now has one authoritative machine record

New missions no longer rely on scattered `for-orchestrator/` files. The orchestrator reads and writes `system/node.json`.

Identity
  • node_id
  • parent_node_id
  • task source
Lifecycle
  • pending / active / waiting_on_computation
  • finished / failed
  • terminal outcome + reasons
Control
  • next_role
  • next_action_after_child_report
Progression
  • current step index
  • steps
  • latest child report
Why this matters
A node is now much easier to inspect, validate, recover, and eventually experiment on because its state is centralized.

Agent Flow

The system is a planner-child-evaluator loop over durable nodes

Agents are temporary contributors. Nodes persist. The orchestrator decides what role runs next.

Plannercreates plan
Worker Childdoes one step
Evaluatorcontinue / replan / wait
More Childrenrepeat as needed
Synthesizerproduces final output

Parent node

holds plan + progression + state
→

Child node

gets goal + context.md + own node.json
→

Parent evaluator

reads child result and updates next action
Extra detail
The current important roles are planner, worker, mid-plan-evaluator, and synthesizer. This flow is now proven in real canaries, not only simulated runs.
