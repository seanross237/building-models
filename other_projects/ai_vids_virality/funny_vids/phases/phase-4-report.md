## Deviations from the plan

| # | Deviation | Reason |
|---|---|---|
| 1 | `backend/main.py` registers a dynamic `@app.get("/data/{file_path:path}")` route handler instead of `app.mount("/data", StaticFiles(directory=DATA_ROOT))` | The tests change the data root per-test via the `AI_VIDS_DATA_ROOT` env var. A static `StaticFiles` mount is bound at app import time and can't follow that — the `test_data_mount_serves_generated_frame` test was failing because the mount was still pointing at the production data dir. A dynamic route that resolves `_data_root()` per request handles both the production case and the per-test case cleanly, and also validates path-traversal at the route level (`target.relative_to(data_root)`) so a browser can't escape the data directory. The route is still registered last, so it can never shadow `/api`. |
| 2 | `backend/main.py` accepts an `AI_VIDS_OFFLINE_STORYBOARD=1` env var that forces the storyboard pipeline to use a null model adapter + stub storyboard adapter | The Phase 1/2/3 API tests walk through `approve_premise`. Without the env var they'd shell out to the real Claude CLI on every `test_approve_premise_*` run — initial local runs took ~40 seconds for that reason. Setting this env var in the test fixtures keeps the offline suite under 1.1s, and production (no env var) still calls the real Claude + Gemini pipeline. |
| 3 | `storyboard/stub_storyboard.py` (legacy Phase 1 module) is kept on disk but is no longer imported anywhere | Deleting it risks breaking a stale import I might have missed. It's 80 lines and doesn't affect any live code path — `backend/main.py` now imports `storyboard.pipeline`, and the new stub lives at `storyboard.adapters.stub_storyboard`. Can delete it in a follow-up sweep if desired. |
| 4 | The beat sheet parser (`storyboard.beats._extract_beats`) requires the LLM response to use the exact `{"premises": [...]}` shape, not a bare list — when the LLM emits a bare list, the parser falls back to a single-beat skeleton | The Phase 3 `find_premise_list` walker is generous but I kept Phase 4 strict on purpose, since the schema contract is more important for beats than for loglines. One of the tests (`test_beats_accept_bare_beats_array`) asserts this fallback behavior explicitly. |
| 5 | `GeminiImageAdapter.generate_frame` only sends `inline_data` parts for character refs whose `name` appears in the current beat's `characters` list | Plan says "passing character ref images as additional input" but not "all of them". Sending only the relevant refs keeps the prompts short, cheap, and focused — the model isn't distracted by characters who aren't in the shot. Covered by `test_gemini_frame_excludes_unrelated_character_refs`. |
| 6 | `storyboard.pipeline.run_storyboard()` exposes a `transition: bool` kwarg (defaulting to True) so tests can generate a storyboard without advancing the sketch state machine | Purely a test ergonomics seam. Production callers leave it at the default. |
| 7 | `storyboard.pipeline.regenerate_frame()` reloads character refs from `storyboard.json` (not from the live sketch object) before regenerating | The sketch object only stores per-beat frame metadata, not per-character refs. The full character ref list lives in `data/sketches/{id}/storyboard.json`. Reading it back guarantees the regen step uses the exact same character anchors as the original run. |
| 8 | The Phase 1 `storyboard/stub_storyboard.py` module is still physically present at the old path | Kept to avoid breaking any external docs or tests that might reference it. All the active code uses the new layout. |

## Storyboard Review page (one paragraph)

The Storyboard Review page now renders one wide card per sketch. The card header shows the sketch ID, the adapter tag ("adapter: stub-storyboard-v1" or "adapter: gemini-2.5-flash-image"), the approved premise logline, and the premise tone, with an "approve storyboard" button and a "reject" button on the right. Below the header is a **row of circular character chips** — one per unique character extracted from the beats, each showing a thumbnail pulled from `/data/sketches/{id}/refs/{slug}.png` and the character's name. Below the chips is a **responsive CSS grid of frame thumbnails**, one per beat (auto-fitting 220px columns). Each cell has the full beat image rendered from `/data/sketches/{id}/storyboard/beat-{NN}-start.png`, a label reading "Beat N · camera_tag", the first sentence of the beat action, meta (location + duration), and a compact "regen this frame" button. Clicking regen POSTs `{"beat": N}` to `/api/sketch/{id}/storyboard/regenerate`, and on success the thumbnail's `src` is cache-busted with `?t=Date.now()` so the new frame renders in place without a full page reload. Frames whose adapter returned an error render with a red border and an "error: ..." line. Approve/reject at the header level use the existing `/api/sketch/{id}/approve_storyboard` and `/reject` endpoints, so the Phase 1/2/3 transitions downstream (video → critic → publish) are untouched. Every other page (Signals, Premise Review, Critic Review, Published, Rejected) still renders exactly as it did at the end of Phase 3 — only the Storyboard Review content changed.
