## Deviations from the plan

| Deviation | Reason |
|---|---|
| `Sketch` is implemented as a `dataclass` instead of a Pydantic model | Pydantic v2 + Python 3.9 + nested mutable defaults required either field validators or a pile of `Optional` boilerplate. The plan explicitly says "dataclass / Pydantic model"; dataclass is simpler and the JSON shape is unchanged, so the seam (`Sketch.to_dict` / `Sketch.from_dict`) is identical for downstream callers. |
| Added `AI_VIDS_DATA_ROOT` env var as the override hook for the FastAPI app | Needed so `tests/test_api.py` and `tests/test_end_to_end.py` can spin up the app against `tmp_path`. The plan says "Spin up Store in a tmp dir" for the integration test; this env var is the cleanest seam to do that without monkey-patching the app at import time. |
| `Status` enum includes `SIGNAL` and `IDEA_PENDING` even though Phase 1 sketches start at `PREMISE_REVIEW` | The plan's enum lists both, and they're useful for the transition table. They're unused by Phase 1 stubs but already wired into `VALID_TRANSITIONS` so Phase 2 (which will create sketches earlier in the flow) doesn't have to retroactively touch the state machine. |
| Did not write a `prompt.md` for the analyzer beyond a placeholder, did not write a `schema.json` enforcement step | Phase 1 analyzer is rule-based and doesn't call an LLM, so the prompt and schema are unused this phase. They exist as files so Phase 2 can drop a real prompt in without changing anything else. |
| Did not implement `idea_pending` queue marker between analyzer and factory | Plan says analyzer promotes to `data/queues/idea-factory/` and factory consumes from there, which is exactly what the implementation does. The `IDEA_PENDING` enum value is reserved for Phase 2 when the controller loop will need an intermediate "claimed by worker" state. No deviation in the file flow, just a forward-looking enum value. |

## UI revision (post-Phase 1)

The original Phase 1 UI was a side-by-side kanban with one column per stage. Per follow-up feedback, `backend/static/index.html` was rewritten as a **multi-page layout**:

- A sticky top nav bar with one button per pipeline stage in left-to-right order: **Signals -> Premise Review -> Storyboard Review -> Critic Review -> Published -> Rejected**, with a visually-separated `+ new idea` button on the far right.
- Each nav button shows a count badge (`Premise Review (3)`) populated from the same `/api/kanban` payload, so the user can see at a glance where work is waiting. No backend changes were required.
- Clicking a nav button switches the main content area to that stage's page; the active button is highlighted with the accent color (border + text + a filled badge).
- Each page renders only the cards for its stage in a centered, full-width vertical list. Cards are bigger now: they show the sketch ID, the logline (16px), a meta row (tone / signal_id / updated_at), the stage-appropriate action buttons (`Approve` / `Reject` / `Publish` — unchanged endpoints), and a `details` toggle that fetches `/api/sketch/{id}` and inlines the full sketch JSON in a collapsible block.
- After any action, the affected card disappears from the current page (its status changed) and the user navigates to the next stage to find it. The detail cache for that sketch is invalidated so the next expand re-fetches.
- Constraints respected: single HTML file, vanilla JS, no build step, no framework, no backend changes. The Signals page renders an empty state explaining that Phase 1 promotes signals straight to premise review and the page will start showing items in Phase 2.
